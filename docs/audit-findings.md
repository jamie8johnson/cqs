## Robustness

#### `QueryCache::get` byte-slice query preview panics on multi-byte chars near offset 40
- **Difficulty:** easy
- **Location:** `src/cache.rs:1021-1028`
- **Description:** On a sqlite read failure, the warn! emission slices `query` by raw byte length: `let preview_len = query.len().min(40); ...query_preview = %&query[..preview_len]`. `query` is the raw user-supplied search string. If it contains multi-byte UTF-8 characters (CJK, emoji, accented Latin) and is longer than 40 bytes, byte 40 may land in the middle of a UTF-8 codepoint and `&query[..40]` panics. The combination is rare but the failure path is precisely "DB error log" — i.e. a degraded query cache state — so a panic here turns a soft failure into hard process death (or in daemon mode, a `catch_unwind` payload). Sister patterns in the same file (cache.rs:175, 216, 232, 260, 286) also use `[..8.min(...)]` but those are on hex strings (ASCII), so safe.
- **Suggested fix:** Replace the manual byte slice with the existing `floor_char_boundary` pattern used by `cli/commands/search/query.rs:50-58` and `embedder/mod.rs:629-634`: `let preview_len = query.floor_char_boundary(40); ... query_preview = %&query[..preview_len]`. `floor_char_boundary` is stable in Rust 1.95 (see `cli/commands/train/task.rs:907` for a production caller) so no wrapper needed.

#### `cli::commands::io::blame::run_git_log` truncates git stderr by raw byte position
- **Difficulty:** easy
- **Location:** `src/cli/commands/io/blame.rs:144-152`
- **Description:** SEC-9 truncates oversized git stderr to 256 bytes via `&stderr[..MAX_STDERR_LEN]`. `stderr` comes from `String::from_utf8_lossy(&output.stderr)`, so it's valid UTF-8 — but invalid input bytes are replaced with U+FFFD (3-byte UTF-8) and any path component the user passed through (e.g. emoji-named files, CJK paths on Windows) ends up in the stderr text. If byte 256 lands inside a multi-byte codepoint, the slice panics. A user with a non-ASCII path that triggers a long git error message hits this. The `format!("{}... (truncated)", &stderr[..256])` then becomes a panic instead of a clean error.
- **Suggested fix:** Use `stderr.floor_char_boundary(MAX_STDERR_LEN)` for the slice end so truncation always lands on a valid UTF-8 boundary. Alternatively use `chars().take(MAX_STDERR_LEN)` (cheaper and more obviously correct) and budget by char count rather than bytes.

#### `LlmClient::sanitize_untrusted` `chars().next().expect("valid UTF-8")` after byte-step iteration
- **Difficulty:** easy
- **Location:** `src/llm/prompts.rs:21-55` (the line-by-line walk over `content.as_bytes()` plus the recovery `let ch = content[i..].chars().next().expect("valid UTF-8")` at line 50)
- **Description:** The function iterates by byte (`while i < bytes.len()`) but reaches into `content` (a `&str`) at byte offset `i` to fetch the next char for output. Today `i` is always advanced by either (a) `ch.len_utf8()` (line 52, which keeps `i` on a char boundary) or (b) `after_lt + prefix.len()` after a sandbox-marker rewrite (line 45) — `prefix.len()` is 17 ASCII bytes and `after_lt` is 1 or 2 ASCII bytes, so this branch also keeps `i` on a char boundary today. The `.expect("valid UTF-8")` is therefore defensible. But the invariant is non-obvious: any future tweak to the byte-skip arithmetic (e.g. extending the prefix, supporting a longer sandbox marker, supporting case where the prefix bytes are matched ignoring case but partially span a multi-byte char from the input — currently impossible because the prefix is pure ASCII) would silently cause `i` to land mid-codepoint and panic. There's no test that exercises non-ASCII input followed by a `<UNTRUSTED_CONTENT` sandbox marker (`tests/llm_*.rs` has none, and the prompt-construction tests in prompts.rs are missing entirely). The function processes attacker-controlled chunk text on every `--improve-docs` / contrastive-summary call.
- **Suggested fix:** Change the walk to `content.char_indices()` and detect the `<` start by char rather than byte; the body is short (~30 lines) and `char_indices` has the same complexity. Failing that, replace the `.expect("valid UTF-8")` with `.unwrap_or_continue` (i.e. `let Some(ch) = content[i..].chars().next() else { i += 1; continue; }`) so a future bug becomes a wrong-output instead of a panic, and add a test with non-ASCII content + sandbox marker.

#### `extract_doc_fallback_for_short_chunk` slices source by `start_byte` from tree-sitter, no boundary guard
- **Difficulty:** easy
- **Location:** `src/parser/chunk.rs:304-306`
- **Description:** `let start_byte = node.start_byte(); let prefix = source.get(..start_byte)?;` — `source.get(..start_byte)` returns `None` if `start_byte` is not a UTF-8 char boundary, which is the right behavior (graceful degrade). But the next line does `prefix.lines()` which fine, BUT: tree-sitter normally emits start_byte at char boundaries for valid UTF-8 source, and if it ever didn't, the function silently returns `None` instead of bubbling a warn. That's correct behavior but worth a `tracing::warn!` so an off-by-one in a future grammar update isn't invisible. More substantively: there is no test for the non-ASCII-source path. A unit test exercising a short Rust function preceded by a comment containing CJK (e.g. `// 注釈\nfn foo() {}`) would lock down the boundary contract.
- **Suggested fix:** Wrap the `source.get(..start_byte)?` with `let prefix = source.get(..start_byte).inspect(|_| ()).or_else(|| { tracing::warn!(start_byte, "tree-sitter byte not on char boundary"); None })?;` — actually simpler: `let prefix = match source.get(..start_byte) { Some(p) => p, None => { tracing::warn!(start_byte, "non-boundary start_byte"); return None; } };`. Add a unit test with non-ASCII leading comment.

#### `parser/markdown::parse_markdown_references` slices `text` at regex byte offset without boundary check
- **Difficulty:** easy
- **Location:** `src/parser/markdown/mod.rs:594-596` (`if match_start > 0 && text.as_bytes()[match_start - 1] == b'!'`)
- **Description:** The image-link skip guard reaches into `text.as_bytes()` at `match_start - 1`. If the character immediately preceding the link in the source is a multi-byte codepoint (e.g. a CJK punctuation mark or em-dash), `text.as_bytes()[match_start - 1]` returns the *last byte of the preceding character*, not the preceding character itself. That byte will not equal `b'!'` (since `!` is ASCII, byte 0x21, and continuation bytes are 0x80-0xBF), so the comparison correctly returns false — no panic, no false positive. But the code is byte-fragile: a future check that compared against a non-ASCII marker character (or that `match_start` could be 1 with a 2-byte char in front, returning a continuation byte) would silently misbehave. Worth a comment locking the assumption ("only safe because comparison byte is ASCII").
- **Suggested fix:** Add a one-line comment: `// Safe to compare a single byte: '!' is ASCII (0x21), so a multi-byte char preceding the link cannot accidentally match.` Or use `text[..match_start].chars().next_back() == Some('!')` which reads more clearly and is robust to future marker changes.

#### Daemon socket request path: silent panic on malformed `args` element gives unhelpful error to client
- **Difficulty:** easy
- **Location:** `src/cli/watch.rs:209-217` (`request.get("args").and_then(|v| v.as_array()).map(|arr| arr.iter().filter_map(|v| v.as_str().map(String::from)).collect()).unwrap_or_default()`)
- **Description:** Daemon decodes args as `Vec<String>` by silently dropping any non-string elements via `filter_map`. A malformed client (e.g. one that sends `{"command": "search", "args": ["query", 42, null]}`) gets `args = ["query"]` and proceeds. The integer and null are silently lost — the search runs against the wrong arg vector and returns wrong results, with no signal to the client that anything was dropped. This is consistent with how clap parses, but unlike clap there's no error message — the client thinks they passed three args and gets the wrong answer. Combined with the silent default of `command = ""` (line 208) when `command` is missing or non-string, this makes daemon protocol bugs hard to diagnose.
- **Suggested fix:** When `args` array contains non-string elements, log a `tracing::warn!` with the dropped indices and emit a daemon error response (`write_daemon_error_tracked(&mut stream, "args contains non-string elements")`). Same for non-string `command`. The cost is one branch per request; the benefit is "client passed nonsense" being a first-class error instead of a silent payload mutation.

#### `cli::commands::index::stats::build_stats_output` `as u32` cast on schema_version may wrap
- **Difficulty:** easy
- **Location:** `src/cli/commands/index/stats.rs:173` (`schema_version: stats.schema_version as u32`)
- **Description:** `stats.schema_version` is `i32` from sqlite. Today it's always `≥ 0` (CURRENT_SCHEMA_VERSION = 20) and the column is constrained by app code, but a corrupted index that somehow ends up with a negative value (manual SQL surgery, sqlite version downgrade, schema migration aborted mid-write) becomes a huge `u32` like 4294967295. The displayed JSON then says `"schema_version": 4294967295` which is misleading — the user thinks they have a corrupted-yet-future schema rather than a corruption red flag. #### `emit_json` (CLI envelope) skips NaN/Infinity sanitization that `write_json_line` (batch) performs — silent panic risk
- **Difficulty:** easy
- **Location:** `src/cli/json_envelope.rs:121-125` (`emit_json`) vs `src/cli/batch/mod.rs:1061-1087` (`write_json_line`) and `src/cli/batch/mod.rs:1030-1051` (`sanitize_json_floats`)
- **Description:** PR #1038 introduced the uniform JSON envelope but the CLI and batch surfaces sanitize floats asymmetrically. `write_json_line` catches NaN/Infinity via a retry-with-`sanitize_json_floats` fallback (and the dedicated TC-7 tests confirm this), so daemon/batch JSONL output never panics on a malformed score. `emit_json` (~50 CLI command call sites, including `cmd_neighbors`, `cmd_diff`, eval/diff_review/where/scout/etc.) just calls `serde_json::to_string_pretty(&env)?` — `serde_json` returns `Err` for NaN/Infinity, which `?` propagates as `anyhow` and `main` prints to stderr. No retry, no sanitize. The `tc13_neighbor_entry_nan_score_becomes_null` test in `src/cli/commands/search/neighbors.rs:262-294` already documents that f32::NAN flows through scoring (e.g. `dot()` on a corrupt embedding, `apply_parent_boost` on overflow); the test currently passes via `to_value`'s silent-null behaviour, but `to_string_pretty` on the same value via `emit_json` errors. Net effect: a NaN that flows through CLI returns `Err` (no JSON written), while the daemon path writes a sanitized envelope. Same input, different observable output. Same class for `chat.rs:225-232` (`wrap_value` + `to_string_pretty` with no sanitize).
- **Suggested fix:** Pull the sanitize-and-retry logic out of `write_json_line` into a private helper next to `emit_json` (or move `sanitize_json_floats` from `batch/mod.rs` to `json_envelope.rs` and reuse). Have `emit_json` and the chat write path both go through that helper. One extra `to_value` round-trip per CLI JSON emit is cheap; correctness parity with the batch surface is the win.

#### `extract_doc_fallback_for_short_chunk` treats `#[derive(...)]`, `#include`, `#define`, `*ptr` as comment-like — leaks attribute/preprocessor/code into doc
- **Difficulty:** easy
- **Location:** `src/parser/chunk.rs:259-276` (`line_looks_comment_like`) called from `src/parser/chunk.rs:289-341` (`extract_doc_fallback_for_short_chunk`); enabled at `src/parser/chunk.rs:88` (`.or_else(...)`)
- **Description:** PR #1040's fallback walks back up to 8 preceding lines, gating each line through `line_looks_comment_like`. That predicate accepts any line whose `trim_start()` begins with `#`, `*`, `--`, `/*`, `*/`, `//`, `<!--`, `<%--`, or `(*`. That's an over-broad classifier:
  - `#[derive(Debug, Clone)]` → `#` → captured as "doc" for the immediately following short Rust struct/enum
  - `#[serde(...)]`, `#[cfg(...)]`, `#[allow(...)]` → same
  - `#include <stdio.h>`, `#define MAX 100`, `#pragma once` → captured as "doc" for the next short C/C++ chunk
  - `#region`, `#endregion` → C# pragmas captured as doc
  - `*ptr = 0;`, `*x = y * z;` (C/C++ pointer deref / multiplication continuation) → `*` → captured as "doc"
  - `/* comment */ code;` is fine, but `*/` followed by trailing actual code? — no, `starts_with("*/")` only fires if the line *starts* with `*/`, so that's narrow.
  The doc is then passed downstream to NL templates and embedded as part of the chunk's natural-language description. For a Rust file dense in `#[derive]` macros (cqs's own `define_chunk_types!`-style code) the embedding now contains the derive list as the chunk's "documentation," reducing semantic precision instead of the truncated-gold lift PR #1040 was added to deliver. None of the existing tests cover the attribute/preprocessor case (`short_chunk_following_code_keeps_doc_none` only exercises a `fn` body — code that doesn't start with `#` or `*`).
- **Suggested fix:** Tighten `line_looks_comment_like` to require evidence of an actual comment, not just "starts with a sigil that *could* be one". Specific changes:
  1. `t.starts_with('#')` → `t.starts_with("# ")` or `t == "#"` (sh/Python) — explicitly reject `#[`, `#!`, `#include`, `#define`, `#region`, `#pragma`. (Note `#!/usr/bin/env` is a shebang and acceptable as a doc, but most short-chunk callers aren't shell scripts.)
  2. `t.starts_with('*')` → `t.starts_with("* ")` or `t == "*"` — accepts the `/** ... */` interior lines while rejecting C statements like `*ptr = ...`.
  3. Add a sad-path test `rust_short_struct_after_derive_keeps_doc_none` with `#[derive(Debug)]\nstruct Tiny { a: u32 }` and a similar `c_short_function_after_include_keeps_doc_none` to pin the new behavior.

#### `tolerated_blanks < 4` magic constant in `extract_doc_comment` — sibling neighbor of named `FALLBACK_DOC_MAX_LINES`
- **Difficulty:** easy
- **Location:** `src/parser/chunk.rs:193,213` (the `4` literal) vs `src/parser/chunk.rs:252,257` (named constants `SHORT_CHUNK_LINE_THRESHOLD`, `FALLBACK_DOC_MAX_LINES`)
- **Description:** PR #1040 added a blank-sibling tolerance loop to `extract_doc_comment` to walk past tree-sitter gap nodes when looking for SQL `--` doc blocks. The cap is hardcoded as `tolerated_blanks < 4` with the rationale ("malformed file with thousands of empty siblings can't make us walk to the start of the file") only in a comment. The same module already promotes two other parser thresholds to named consts (`SHORT_CHUNK_LINE_THRESHOLD = 5`, `FALLBACK_DOC_MAX_LINES = 8`), so the inconsistency is jarring on a single read of the file and the magic number gives no test or doc anchor when the next person bumps it to 6 or 8. (Bumping the value is plausible: the F# `(*  *)` block comment grammar could emit more whitespace siblings between docstring and definition than the SQL grammar.)
- **Suggested fix:** Add `const MAX_TOLERATED_BLANK_SIBLINGS: usize = 4;` next to `SHORT_CHUNK_LINE_THRESHOLD` (with the same "preceding-blank-sibling cap" rationale) and replace the `< 4` literal at line 213 with `< MAX_TOLERATED_BLANK_SIBLINGS`.

#### `Store::open_readonly_pooled` and `open_readonly_pooled_with_runtime` duplicate `StoreOpenConfig` literal — drift risk
- **Difficulty:** easy
- **Location:** `src/store/mod.rs:694-706` (`open_readonly_pooled`) vs `src/store/mod.rs:733-748` (`open_readonly_pooled_with_runtime`); contrast with `src/store/mod.rs:664-681` (`default_open_config`) which already centralizes the read-write pair (`open` and `open_with_runtime`)
- **Description:** The read-write side of `Store` already factors its defaults into a shared `default_open_config` builder so `open` and `open_with_runtime` cannot drift on `max_connections`, `mmap_size`, `cache_size`, or future fields. The read-only pooled side duplicates the `StoreOpenConfig { read_only: true, use_current_thread: true, max_connections: 1, mmap_size: resolve_mmap_size("268435456", path), cache_size: cache_size_from_env("-16384"), runtime: ... }` literal twice — once for the daemon-runtime variant, once for the standalone variant. Add `fn default_readonly_pooled_config(path: &Path, runtime: Option<Arc<Runtime>>) -> StoreOpenConfig` to `impl Store<ReadOnly>` returning the shared literal with `runtime` plugged in, then have both `open_readonly_pooled` and `open_readonly_pooled_with_runtime` delegate to it. While there, consider whether `open_readonly` and `open_readonly_small` should also share a `default_readonly_minimal_config` builder — they're only differentiated by a single arg (mmap_size + cache_size) so the duplication is currently lower-risk, but the same drift pattern applies.

#### `resolve_splade_alpha` global env arm silently swallows malformed/non-finite values; per-cat arm warns
- **Difficulty:** easy
- **Location:** `src/search/router.rs:449-464` (global env match) vs `src/search/router.rs:427-446` (per-cat env match)
- **Description:** The per-category branch warns on parse failure (`"Invalid alpha, using default"`) and on non-finite (`"Non-finite alpha, using default"`). Operators tuning the global knob to debug a regression get no feedback that their value was rejected; they assume the alpha was applied and look for the bug somewhere else. OB-NEW-1 (already wave-1) added the success-path log; the failure-path divergence is the leftover.
- **Suggested fix:** Mirror the per-cat arms. Add `Ok(val) => tracing::warn!(var = "CQS_SPLADE_ALPHA", value = %val, "Invalid alpha, using default");` and inside the happy-path arm add the same `tracing::warn!(var = "CQS_SPLADE_ALPHA", value = %val, "Non-finite alpha, using default");` after the `if alpha.is_finite()` block, so the two arms stay in lockstep.

#### `handle_socket_client` 5s read / 30s write hardcoded — unaddressed `TODO(cross-coordination)` wave-1A leftover
- **Difficulty:** easy
- **Location:** `src/cli/watch.rs:153,159` (the hardcoded `Duration::from_secs(5)` and `Duration::from_secs(30)`) and `src/cli/dispatch.rs:634-636` (the explicit TODO referring to it)

Call it from both `handle_socket_client` (lines 153, 159) and `try_daemon_query` (line 637). #### `wrap_value`/`wrap_error` (json! macro) duplicate `Envelope::ok`/`Envelope::err` (typed) — two implementations of one shape
- **Difficulty:** easy
- **Location:** `src/cli/json_envelope.rs:71-98` (typed `Envelope::ok` / `Envelope::err`) vs `src/cli/json_envelope.rs:100-117` (untyped `wrap_value` / `wrap_error`)
- **Description:** PR #1038 added two parallel implementations of the envelope shape: the typed `Envelope<T>::ok(data)` / `Envelope<serde_json::Value>::err(code, message)` (used by tests today, `#[allow(dead_code)]` on `err`), and the untyped `wrap_value` / `wrap_error` (`serde_json::json!` macros). Both produce `{data, error, version}` but via different code paths. Today they emit identical JSON, but if the envelope shape ever evolves — e.g., adds a `meta: Option<...>` for trace-ids or a `partial: bool` for streaming — only one will get updated and the two surfaces will diverge. This is the exact failure pattern of the `dispatch_scout` JSON-shape divergence in CQ-V1.25-7 (still pending). It also compounds CONTRIBUTING.md's already-confused error-codes documentation (the `#[allow(dead_code)]` on `Envelope::err` flagged as a separate Documentation finding above).
- **Suggested fix:** Make `wrap_value` and `wrap_error` thin wrappers over the typed path. `wrap_value(payload) → serde_json::to_value(Envelope::ok(payload)).unwrap_or_else(|e| { tracing::warn!(error=%e, "envelope serialize failed"); serde_json::json!({"data": payload, "error": null, "version": JSON_OUTPUT_VERSION}) })` (the fallback only matters if `Envelope` ever gets a non-Serializable field, which today is impossible — so the unwrap is structurally safe). `wrap_error(code, message) → serde_json::to_value(Envelope::<serde_json::Value>::err(code, message)).unwrap()`. Drop `#[allow(dead_code)]` from `Envelope::err` once it has the `wrap_error` caller. One source of truth, one place to bump on schema changes.

## API Design

#### `cqs --json eval queries.json` does not emit JSON — top-level `--json` cascade missing
- **Difficulty:** easy
- **Location:** `src/cli/dispatch.rs:495` (dispatch passes `args` only) and `src/cli/commands/eval/mod.rs:95-99,112` (handler reads `args.json` only, never `ctx.cli.json`)
- **Description:** `cqs eval` is a recently-added CLI command. The dispatch site routes `cmd_eval(&ctx, args)` without OR-ing the top-level `cli.json` into `args.json`. CONTRIBUTING.md and the dispatch.rs comment at line 206-212 document the convention: "every subcommand reads `cli.json || output.json`. Top-level `--json` wins". Eval breaks that — `cqs --json eval queries.json` produces text output, while every other subcommand honors the top-level flag. `cmd_eval` similarly reads `args.json` directly at line 95 and passes `args.json` to `print_diff_report` at line 112. An agent doing `cqs --json eval ... | jq` gets an unparseable text report and has no way to know which subcommands honor the precedence rule.
- **Suggested fix:** Take advantage of the already-passed `&CommandContext` and resolve `let json = ctx.cli.json || args.json;` once at the top of the handler, then pass `json` through to `print_text_report` / `emit_json` / `print_diff_report`. This matches the pattern used by `cmd_notes_list` (`if json || ctx.cli.json` at `src/cli/commands/io/notes.rs:524`) and `cmd_model_show` (`cli.json || *json` at `src/cli/commands/infra/model.rs:113`).

#### `cqs --json project search "foo"` and `cqs --json cache stats` ignore top-level `--json`
- **Difficulty:** easy
- **Location:** `src/cli/dispatch.rs:69` (`cmd_cache(subcmd)` — no `cli`) and `src/cli/dispatch.rs:147-149` (`cmd_project(subcmd, ...)` — no `cli`); handlers at `src/cli/commands/infra/project.rs:67-156` and `src/cli/commands/infra/cache_cmd.rs:35-46`
- **Description:** Same precedence violation as Eval. `cmd_model` (`src/cli/commands/infra/model.rs:113-119`) takes `cli` and OR-s `cli.json || *json`, which is the documented pattern. `cmd_project` and `cmd_cache` don't take `cli` and so each subcommand reads only its own `--json` flag. `cqs --json project search "regex"` and `cqs --json cache stats` both return text. An agent that wired `cqs --json` into a wrapper relying on the documented "top-level always wins" rule has two silent gaps.
- **Suggested fix:** Change both signatures to take `cli: &Cli` (model already does this — see line 110: `cmd_model(cli: &Cli, subcmd: &ModelCommand)`). At each call site OR `cli.json || *json`. Add an integration test (`tests/cli_commands_test.rs` style) asserting `cqs --json cache stats` and `cqs --json project search "x"` produce JSON envelopes, so the convention can't silently regress on the next subcommand added.

#### `cqs doctor --json` always prints text checks before the JSON document
- **Difficulty:** easy
- **Location:** `src/cli/commands/infra/doctor.rs:97-322` (handler runs ~290 lines of `println!` text output unconditionally, then maybe emits JSON at line 316); flag spec at `src/cli/definitions.rs:304-308`
- **Description:** `cmd_doctor` runs the full text check sequence first — colored runtime / parser / index / HNSW / cuVS sections, ~93 `println!` calls — and only afterward emits the JSON envelope at line 316 when `json` is set. So `cqs doctor --json` produces a stream that starts with `Runtime:\n  [✓] Model: ...` and only after the entire human-readable report ends does the `{"data": ...}` envelope appear. `jq`/`json.loads` cannot consume this. The flag's clap doc at `src/cli/definitions.rs:304-308` says `--json` "Emit `--verbose` output as JSON instead of text" — but the implementation emits both. Compounding the confusion, the doc at line 306 says "Ignored without `--verbose` (the default check format is text-only)" while the implementation at line 112 explicitly inverts that ("`--json` implies `--verbose`"). So both the doc and the behaviour are mutually inconsistent.
- **Suggested fix:** When `json` is set, suppress the human checks entirely (gate the per-section `println!` blocks on `!json`) and emit one JSON document covering both the check results and the verbose introspection. Update the clap doc to match the implementation. If keeping the colored checks is valuable for human use of `--json` from a TTY, route them to stderr — stdout becomes pure JSON, stderr stays human-friendly, and `cqs doctor --json | jq` works.

#### `cqs ping --json` emits text on the no-daemon error path — JSON consumers get nothing on stdout
- **Difficulty:** easy
- **Location:** `src/cli/commands/infra/ping.rs:182-188` (Err arm calls `eprintln!("cqs: {msg}")` and `std::process::exit(1)` even when `--json` was requested); same pattern at `src/cli/commands/eval/mod.rs:117-122`
- **Description:** When `--json` is set and the daemon is not running, `cmd_ping` exits with code 1 and prints `cqs: <msg>` to stderr. Stdout is empty. The whole point of the JSON envelope shipped in PR #1038 (and documented in `CONTRIBUTING.md:79-84` with `not_found`/`io_error` codes) is exactly this case — agents inspect `error.code` instead of parsing stderr text. `cqs ping` is the closest analogue to a healthcheck and is the most likely consumer of structured error output. The same shape recurs in `cqs eval --baseline X.json --json`, where regression past tolerance writes `[eval] N regression(s) ...` to stderr and `exit(1)` instead of an enveloped error on stdout.
- **Suggested fix:** When `json && Err(...)`, call `crate::cli::json_envelope::emit_json_error(error_codes::NOT_FOUND, &msg)?;` (or `IO_ERROR` for transport failures) before `process::exit(1)`. This activates the `#[allow(dead_code)]` `emit_json_error` helper at `src/cli/json_envelope.rs:130` for the first time and gives a real production usage of the `not_found` / `io_error` codes that CONTRIBUTING.md advertises. Same change for `cqs eval`'s regression exit path.

#### Three different "dry-run vs apply" idioms across mutating commands
- **Difficulty:** medium
- **Location:** `src/cli/definitions.rs:296` (`Doctor --fix`), `src/cli/definitions.rs:619` (`Convert --dry_run`), `src/cli/args.rs:435` (`Suggest --apply`), `src/cli/args.rs:544` (`Index --dry_run`)
- **Description:** Four mutating-or-could-be-mutating commands use three different conventions for "default safe vs default mutate":
  - `Doctor`: default is read-only checks; `--fix` opts in to mutation. (Default safe.)
  - `Suggest`: default is dry-run; `--apply` opts in to writing notes.toml. (Default safe.)
  - `Convert`: default writes Markdown files; `--dry_run` opts out. (Default mutates.)
  - `Index`: default writes the index; `--dry_run` opts out. (Default mutates.)
  An agent has to remember per-command which side is default — there is no single rule. The `BatchSupport` classifier at `src/cli/definitions.rs:828-838` already has special-case logic to inspect `args.apply` because the convention is non-uniform. New mutating commands will pick a fourth or fifth idiom unless this is settled.
- **Suggested fix:** The split between "side-effect commands default to mutating" (`index`, `convert`) and "analyser commands default to read-only" (`doctor`, `suggest`) is defensible — but it needs to be explicit. Add a paragraph to CONTRIBUTING.md documenting the rule. Also fold the per-command intent into the help text: `Index --dry-run` could read "Show what would be indexed (default writes the index)" rather than just "Show what would be indexed, don't write". Cheaper than renaming flags and breaks no scripts.

#### `cqs eval` JSON output uses `r1`/`r5`/`r20` while sibling `EvalReport` uses `r_at_1`/`r_at_5`/`r_at_20`
- **Difficulty:** easy
- **Location:** `src/cli/commands/eval/baseline.rs:28-33` (`KDelta { r1, r5, r20 }`) vs `src/cli/commands/eval/runner.rs:74-76, 83-85` (`StatsBlock { r_at_1, r_at_5, r_at_20 }`); `Regression::metric` value at `src/cli/commands/eval/baseline.rs:37-44` uses third spelling `"R@1"` / `"R@5"` / `"R@20"`
- **Description:** A single `cqs eval` invocation can emit two different JSON shapes. `cqs eval --json` outputs a report whose `overall.r_at_1` matches the conventional `r_at_1` field name. `cqs eval --baseline X.json --json` outputs a `DiffReport` whose per-K deltas use the abbreviated `r1` / `r5` / `r20` field names. An agent that learned the field name from the basic eval and then switched to the diff path will fail to find the metric. Same envelope, same command, different shape.

Recommend `r_at_1`-style for keys (matches the rest of the codebase per CONTRIBUTING.md's snake_case rule) and keep `R@K` strings inside the `metric` value for human display since that's a label, not a key.

#### `cqs cache stats --json` returns `total_size_mb` as a string, violating numeric-output convention
- **Difficulty:** easy
- **Location:** `src/cli/commands/infra/cache_cmd.rs:54-61`
- **Description:** Inside the cache stats JSON object: `"total_size_mb": format!("{:.1}", stats.total_size_bytes as f64 / 1_048_576.0)` — the field's value is a string like `"42.5"`, not a number `42.5`. CONTRIBUTING.md's "JSON Output Field Naming Conventions" (lines 93-108) doesn't explicitly say "numeric fields stay numeric" but the principle is implicit: `score: f32`, `line_start: u32`, `total_chunks: u64` are all numbers across every other JSON output. `cqs model show --json` correctly emits `"index_db_size_bytes": 12345` (number) — agents have no reason to expect cache to differ. Programmatic consumers calling `obj["total_size_mb"] + 1` get a TypeError instead of a numeric add.
- **Suggested fix:** Drop `total_size_mb` entirely (consumers can divide by 1,048,576) or emit it as `f64`: `"total_size_mb": stats.total_size_bytes as f64 / 1_048_576.0` (no `format!`). Update the test in cache_cmd.rs to assert it's a number. This also brings the field in line with the `index_db_size_bytes` / `hnsw_size_bytes` convention from `cqs model show` — keep `_bytes` suffix on raw counts and let consumers do unit conversion.

#### `dispatch_line` and `cmd_batch` main loop emit envelope errors with no `tracing::warn!` — daemon fails silently in journal
- **Difficulty:** easy
- **Location:** `src/cli/batch/mod.rs:466-510` (`dispatch_line` — the daemon socket entry path) and `src/cli/batch/mod.rs:1306-1401` (the `cmd_batch` stdin loop), specifically lines 476, 502, 507, 1311, 1333, 1358, 1377, 1390 — all eight `write_envelope_error(...)` sites
- **Description:** PR #1038 introduced the envelope error pipeline, but every error site in the daemon dispatch path bumps `error_count.fetch_add(1, Ordering::Relaxed)` and then writes the envelope error to the client with no corresponding `tracing::warn!` or `tracing::error!`. The handler-internal serialize-failure fallback at `src/cli/batch/mod.rs:1075` has tracing (`"JSON serialization failed after sanitization"`), so the rest of the file knows the pattern — it just wasn't applied to the dispatcher itself. Result: a daemon serving a misbehaving client (parse errors at 100/s, anyhow chain failures from a CUDA OOM, null-byte injection attempts caught by RT-INJ-2 at line 1331) shows the rising `error_count` via `cqs ping` but writes nothing to the systemd journal — a 3am page-out engineer can see "errors went up" but not "what failed and why". The daemon socket handler at `src/cli/watch.rs:330-335` does log per-query `status` and `latency_ms`, but it can only report `"ok"` / `"client_error"` / `"panic"` from the outer dispatch result — the actual `error_codes::INTERNAL` vs `PARSE_ERROR` vs `INVALID_INPUT` discrimination already encoded at the envelope-error site is invisible from there.

The CUDA-OOM case (line 502) is the most load-bearing — that's the one users hit in practice. Bonus: extract `fn report_dispatch_error(out, code, message, &e)` that does both the write and the log so future call sites can't drift back to silent.

#### `cmd_doctor --fix` logs `tracing::warn!("cqs index exited with status {}", status)` — unstructured, status not parseable
- **Difficulty:** easy
- **Location:** `src/cli/commands/infra/doctor.rs:59` and `src/cli/commands/infra/doctor.rs:75`
- **Description:** Both `tracing::warn!("cqs index exited with status {}", status)` (line 59) and `tracing::warn!("cqs index --force exited with status {}", status)` (line 75) interpolate `std::process::ExitStatus`'s `Display` impl into the message string. ExitStatus prints as e.g. `"exit status: 1"` or `"signal: 15 (SIGTERM)"`, which makes structured filtering (`grep '"code":1'` or any structured journal query) impossible. #### `extract_doc_fallback_for_short_chunk` allocates `Vec<&str>` over entire prefix per short chunk — O(N²) parse-time on file with many short chunks
- **Difficulty:** easy
- **Location:** `src/parser/chunk.rs:289-322` (function), specifically line 306 (`let lines: Vec<&str> = prefix.lines().map(|l| l.trim_end_matches('\r')).collect();`)
- **Description:** PR #1040's parse-time fallback splits the entire source prefix from start-of-file to chunk start_byte into a `Vec<&str>` on every short-chunk extraction. The cost: for a chunk at byte offset B in a file with line-density D, the Vec holds B/D entries (~80 bytes each on 64-bit). The fallback runs per-chunk via `extract_chunk` (see `chunk.rs:88` `.or_else(...)`). For a SQL file with 200 small `CREATE TABLE` definitions distributed evenly, the fallback fires 200 times — chunk 1 splits 0 lines, chunk 100 splits ~1500 lines, chunk 200 splits ~3000 lines. Total work: O(M²) where M is the chunk count, plus 200 separate Vec allocations of growing size. For a corpus where 30%+ of chunks are short (the truncated_gold population PR #1040 targets — concentrated in SQL/markdown/short-helper files), this becomes the dominant parse-time cost. The split is also wasteful: only the last `FALLBACK_DOC_MAX_LINES = 8` lines are ever inspected, but the entire prefix is split first.
- **Suggested fix:** Walk the prefix in reverse from `start_byte` only as far back as 8 newlines, then split that small slice. Concrete:
  ```rust
  // Find the byte position of the 8th-from-last newline in the prefix.
  let prefix = source.get(..start_byte)?;
  let bytes = prefix.as_bytes();
  let mut newline_count = 0;
  let mut tail_start = bytes.len();
  for i in (0..bytes.len()).rev() {
      if bytes[i] == b'\n' {
          newline_count += 1;
          if newline_count > FALLBACK_DOC_MAX_LINES + 1 { break; }
          tail_start = i + 1;
      }
  }
  let tail = &prefix[tail_start..];
  let lines: Vec<&str> = tail.lines().map(|l| l.trim_end_matches('\r')).collect();
  ```
  This bounds the per-chunk work to O(8 lines) regardless of file size. Existing unit tests at `chunk.rs:1164+` validate the behavior; add `parses_500_short_chunks_in_one_file` (e.g. a synthesized `CREATE TABLE` × 500 SQL fixture) as a smoke test that the parse stays linear in chunk count, not quadratic.

## Algorithm Correctness

#### `BoundedScoreHeap::push` evicts the *best* tied-score id instead of the *worst* — non-deterministic top-K under HashMap-fed input
- **Difficulty:** medium
- **Location:** `src/search/scoring/candidate.rs:193-217` (`push`) and `src/search/scoring/candidate.rs:225-233` (`into_sorted_vec`); used by `Store::rrf_fuse` (`src/store/search.rs:201-205`) which iterates a `HashMap` whose order is process-seed-randomized.
- **Description:** The doc comment at line 188-192 promises "the surviving set is deterministic" under tie-breaker "smaller id wins". The actual implementation gets this wrong because the min-heap stores `Reverse<(score, id)>`. Final sort order is `(score desc, id asc)`, so the *worst* element under that order is `(score min, id max)` — which is what should be evicted. But `BinaryHeap::peek()` on `Reverse<(score, id)>` returns the *smallest* tuple = `(score min, id min)` — i.e., the **best** element among the lowest-scored entries. The eviction condition at line 211 (`score == *min_score && id < *min_id`) then evicts that "best at the boundary" instead of the "worst at the boundary". **Concrete failing case** with `capacity=2`:
  - Push order `("c", 0.5), ("b", 0.5), ("a", 0.5)`: after the first two, heap = `[(0.5,"c"), (0.5,"b")]`. `peek()` = `Reverse((0.5,"b"))` (smallest tuple wrapped). On `("a", 0.5)`: `"a" < "b"` true → pop+push. **`"b"` is evicted** (the smallest tuple gets popped from a min-Reverse-heap — heap.pop returns the *largest* `Reverse` which is the *smallest* underlying tuple). Final heap = `[(0.5,"c"), (0.5,"a")]`. `into_sorted_vec` returns `[("a",0.5), ("c",0.5)]`.
  - Push order `("a", 0.5), ("b", 0.5), ("c", 0.5)`: peek after first two = `Reverse((0.5,"a"))`. On `("c", 0.5)`: `"c" < "a"` false → reject. Returns `[("a",0.5), ("b",0.5)]`.
  - Both push orders should produce `{"a","b"}` (the deterministic top-2 by `(score desc, id asc)`); the second order does, the first returns `{"a","c"}`.
  - `rrf_fuse` feeds the heap from `for (id, score) in scores` where `scores: HashMap<&str, f32>` — HashMap iteration order is process-seed-randomized, so the bug triggers non-deterministically whenever the candidate pool has tied RRF scores at the limit boundary. This is exactly the AC-V1.25-1 / AC-V1.25-2 class of regression that wave-1 was supposed to close. The existing `test_bounded_heap_equal_scores` at lines 314-326 only exercises the `(a,b,c)` push order so it never trips the bug.
- **Suggested fix:** Make the heap key reflect the actual eviction order. Wrap the id in `Reverse` too: store `Reverse<(OrderedFloat, Reverse<String>)>` so `peek()` returns the worst element under the final sort order (smallest score, *largest* id). #### `Store::search_by_name` tie-breaker uses chunk id which sorts line numbers lexicographically (line "10" < line "2")
- **Difficulty:** easy
- **Location:** `src/store/search.rs:138-148` (`search_by_name` re-sort) — chunk id format defined at `src/parser/chunk.rs:107` (`format!("{}:{}:{}", path, line_start, hash_prefix)`)
- **Description:** When two chunks tie on `score_name_match_pre_lower` (very common — both are exact matches for `name`), the tie-breaker at line 144 (`a.chunk.id.cmp(&b.chunk.id)`) compares chunk ids lexicographically. Chunk ids are formatted as `{path}:{line_start}:{hash_prefix}`. The `line_start` portion is a **base-10 stringified `u32`** with no zero-padding — so within the same file, `"file.rs:10:abc..."` sorts BEFORE `"file.rs:2:def..."` lexicographically (`"1" < "2"` byte-wise). That means when the same `name` is defined at both line 2 and line 10 of one file (e.g., a duplicate `pub fn foo()` in conditional cfg blocks, or a re-export in a `#[cfg(test)]` module), the tie-breaker prefers line 10 — the wrong one for "where is this defined?" semantics. **Concrete failing case** for a Rust file:
  ```rust
  // line 2:  pub fn build() -> Builder { ... }   ← real definition
  // line 10: pub fn build() -> Builder { unimplemented!() }   ← test stub
  ```
  `cqs --name-only build` returns the line-10 stub first, despite the user clearly wanting the line-2 real definition. Cross-file the lex order is also surprising: paths starting with shorter strings sort before longer ones with the same prefix (`"src/a.rs:..." < "src/aa.rs:..."`). The doc comment at lines 138-145 advertises "deterministic order" — true, just not the **right** order.
- **Suggested fix:** Replace the secondary sort key with a tuple that orders the way users expect:
  ```rust
  results.sort_by(|a, b| {
      b.score
          .total_cmp(&a.score)
          .then(a.chunk.file.cmp(&b.chunk.file))    // shorter / alphabetic-earlier file first
          .then(a.chunk.line_start.cmp(&b.chunk.line_start))   // numeric line order
          .then(a.chunk.id.cmp(&b.chunk.id))        // hash for absolute determinism
  });
  ```
  Add a regression test `search_by_name_prefers_earlier_line_in_same_file` building a file with `build` at line 2 and line 10 and asserting line 2 ranks first. Same fix should be propagated to any other sort that uses raw chunk-id as the tie-breaker (grep for `chunk.id.cmp`).

#### HNSW save: backup `std::fs::rename` failure is logged-and-continued, leaving rollback path with no original to restore
- **Difficulty:** medium
- **Location:** `src/hnsw/persist.rs:407-419` (backup loop) and `src/hnsw/persist.rs:457-479` (rollback)
- **Description:** The HNSW save loop renames `final_path` → `final_path.bak` for each extension before invoking `atomic_replace(temp, final)`. If the backup rename fails (line 411: `std::fs::rename`), the code only emits `tracing::warn!` and continues to the atomic_replace step (line 431), which overwrites the *original* `final_path` with the temp file. If `atomic_replace` then fails for a later extension, the rollback path (line 463-475) iterates `.bak` files to restore originals — but no `.bak` was ever created for the file that overwrote silently, so its original is lost. Concrete failing scenario: `dir.join("idx.hnsw.graph")` is held by a concurrent reader on Windows or WSL `/mnt/c` (advisory locking won't actually block the rename on those FSes), so the backup rename fails for `.graph`, succeeds for `.data`/`.ids`/`.checksum`. atomic_replace overwrites `.graph` with the new content. atomic_replace then fails on `.data` (disk full mid-fsync). Rollback removes the new `.data` and restores `.data.bak` — but cannot restore the original `.graph`, so the index is left with a NEW `.graph` paired with the OLD `.data`+`.ids`+`.checksum`. The on-load checksum verifier will reject this combination, but the user has now lost their working index entirely and the only recovery is `cqs index --force`.
- **Suggested fix:** Either propagate the backup-rename error (return Err) so the caller never starts the atomic_replace pass, or skip the atomic_replace for any extension whose backup failed. Easiest is the first: change line 411 to `?` propagation and remove the `if let Err` swallow, since a backup-rename failure on a file we're about to overwrite is a strict precondition violation. Optionally also fsync each .bak rename (currently bare `std::fs::rename` without parent-dir fsync — adds <1 ms per file).

#### `acquire_index_lock` stale-lock removal races with a genuinely-running peer holding the same lock file inode
- **Difficulty:** medium
- **Location:** `src/cli/files.rs:140-179`
- **Description:** When `try_lock` fails, `acquire_index_lock` reads the lock file's PID, calls `process_exists(pid)`, and if false invokes `std::fs::remove_file(&lock_path)` then loops to retry. Three race windows can produce two writers holding "the lock":
  1. PID lookup is approximate on Linux (zombies / PID namespaces / pid recycling). `process_exists(stale_pid)` returns false even though a freshly-spawned `cqs index` process *just* claimed that PID. We remove their lock file inode, then `open_lock_file` creates a new inode at the same path, both processes successfully `try_lock` their respective inodes (the kernel locks per-inode, not per-path), and now two writers run concurrently against the same DB.
  2. Between `process_exists` and `remove_file`, the genuine owner could have crashed and been replaced by a *new* `cqs index` invocation that acquired the lock cleanly. We delete *their* lock file. Subsequent `open_lock_file` creates a fresh inode the new owner doesn't hold a lock on. We acquire it. Two writers.
  3. On WSL `/mnt/c` (NTFS over 9P, per the `warn_wsl_advisory_locking` warnings already in this codebase), `flock` is purely advisory. `process_exists` is unreliable across Windows-side cqs invocations launched via `powershell.exe` — they have Windows PIDs that look completely absent to the WSL-side check.
  Concrete failing scenario: parallel `cqs index` + `cqs watch` reindex on WSL, the watch service is restarting, and the brief window where its PID reads as "not found" coincides with `cqs index` retrying. Both processes then write to the chunks table and HNSW files concurrently, producing schema_version conflicts, mixed embedding dimensions, or torn HNSW graphs.
- **Suggested fix:** Replace the unlink-and-retry approach with a *non-destructive* re-lock attempt: drop the failed `lock_file`, reopen, `try_lock` again. If still busy after the retry, keep the file in place and bail with a clearer "another process holds the lock; PID N may be stale — manually delete .cqs/index.lock if confident it is" message. Removing the lock file inode is the dangerous primitive; the actual safety guarantee comes from per-inode flocking, not from the file's existence.

#### `cmd_telemetry_reset` non-atomic copy-then-truncate can lose entire telemetry log on crash mid-reset
- **Difficulty:** easy

#### Daemon socket response triple-handles bytes: dispatch → UTF-8 validate → re-wrap as JSON-string-in-JSON
- **Difficulty:** medium
- **Location:** `src/cli/watch.rs:281-301` (the `dispatch_line(&full_line, &mut output)` → `String::from_utf8(output)` → `serde_json::json!({"status": "ok", "output": output.trim_end()})` → `writeln!(stream, "{}", resp)` chain)
- **Description:** Every daemon query response is serialized 3 times and copied/wrapped 4 times: (1) `dispatch_line` → `write_json_line` produces JSON bytes into `Vec<u8> output`, (2) `String::from_utf8(output)` validates and takes ownership of those bytes, (3) `serde_json::json!({..., "output": output.trim_end()})` embeds the entire JSON document AS A STRING field of an outer JSON envelope (escaping every `"`, `\`, `{`, `}`, newline inside it), (4) `writeln!(stream, "{}", resp)` re-serializes the outer envelope and writes to the socket. For a search response with 50 results (~50KB JSON), step (3) roughly doubles output size due to escape characters — backslashes for every internal quote — so 100KB of socket bytes for 50KB of payload. Cost scales linearly with response size and runs on every daemon query. The double-encoding also means the client must `serde_json::from_str(stream)` then `serde_json::from_str(parsed["output"])` — two parses for one logical response. This is orthogonal to the existing `wrap_value(value.clone())` finding in Resource Management (which is the inner envelope clone); this is the outer string-in-string envelope.
- **Suggested fix:** Either (a) write the dispatched JSON bytes directly to the socket with a single status header line (e.g. `status: ok\n` followed by the dispatch_output bytes), or (b) make the inner dispatch return a `serde_json::Value` and embed it as a real JSON field rather than a string: `serde_json::json!({"status": "ok", "output": parsed_value})`. Option (b) requires plumbing — `dispatch_line` currently writes to an `impl Write` — so define a sibling `dispatch_value` that returns the `Value` and have the daemon path use that. Saves one parse on the client + one re-encode on the server + the escape-char inflation.

- **Difficulty:** easy
- **Location:** `src/cli/watch.rs:1206-1220` (`UnixListener::bind` then `set_permissions(0o600)` in two separate syscalls), `src/cli/files.rs:13-15` (the SECURITY.md claim "socket created with mode 0o600 so only the owning user can connect")
- **Description:** The socket is created via `UnixListener::bind` (line 1206), which honors the process umask. Concretely: `let canonical = dunce::canonicalize(&file_path).context("Invalid path")?; let project_canonical = dunce::canonicalize(root).context("Invalid project root")?; if !canonical.starts_with(&project_canonical) { bail!("Invalid path"); } if !file_path.exists() { bail!("Invalid path"); }`. Single error class for any "we won't read this", no leak. Same fix applies to similar shape in `cli/commands/search/query.rs:864-875`.

## Platform Behavior

#### `convert::is_safe_executable_path` only blocks `/tmp/` `/var/tmp/` — Windows `%TEMP%` `python.exe` passes the gate
- **Difficulty:** medium
- **Location:** `src/convert/mod.rs:142-165` (the `DANGEROUS_PREFIXES = ["/tmp/", "/var/tmp/"]` slice and the `#[cfg(unix)]` writability check immediately below).
- **Description:** SEC-V1.25-10 hardened `find_python` against PATH-injection by rejecting any binary that resolves into a user-writable directory. The implementation has two prongs: (a) an explicit prefix list `["/tmp/", "/var/tmp/"]`, and (b) a unix-only stat of the parent directory's mode bits. Neither prong fires on Windows. Concretely: a Windows attacker who drops `python.exe` into `C:\Users\<user>\AppData\Local\Temp\` (the documented `%TEMP%`) and prepends that directory to PATH gets a clean pass — `s.starts_with("/tmp/")` is false (Windows paths don't have a leading `/`, plus `to_string_lossy()` of a canonicalized Windows path uses `\` separators), and the `#[cfg(unix)]` block never executes. The `cqs convert` doc-conversion path then runs the attacker's `python.exe` against an attacker-controlled `pdf_to_md.py` (or even the bundled `scripts/pdf_to_md.py`) with the user's full filesystem privileges. PR target list explicitly includes Windows x86_64; this finding is not theoretical for that platform.
- **Suggested fix:** Add Windows-equivalent dangerous prefixes derived at runtime from `std::env::temp_dir()` and `dirs::cache_dir()`. Use `Path::starts_with()` (which compares components, not raw substrings) instead of `to_string_lossy().starts_with()`. For the writability check, on Windows fall back to checking the directory's NTFS DACL via `windows::Win32::Security` (or, simpler, refuse any directory under `temp_dir()`/`cache_dir()`/`%LOCALAPPDATA%\Temp` without trying to inspect ACLs). At minimum, on non-unix platforms, reject any python interpreter that doesn't live under one of a configured allowlist (e.g. `C:\Python*`, `C:\Program Files\Python*`, `%LOCALAPPDATA%\Programs\Python\*`).

#### `aux_model::is_path_like` treats `C:\Models\splade` as an HF repo id, not a filesystem path
- **Difficulty:** easy
- **Location:** `src/aux_model.rs:117-119` (`fn is_path_like(raw: &str) -> bool { raw.starts_with('/') || raw.starts_with("~/") }`) called from `src/aux_model.rs:354` (`resolve_raw`)
- **Description:** The path-vs-repo classifier accepts only POSIX-style absolute (`/...`) and home-relative (`~/...`) prefixes. On Windows, `C:\Models\splade-code-0.6B`, `D:\onnx\reranker\`, or even `\\server\share\splade` all return `false` and are routed through the HF-repo branch. For SPLADE that produces a confusing failure later down the resolve chain (the input is treated as a repo id but SPLADE has no Hub-fetch path); for the reranker, the input is wrapped as `repo: Some("C:\\Models\\splade")` and the HF Hub fetch is attempted with that as a repo id — a request that produces a confusing 404 from the HF Hub. Add unit tests pinning `is_path_like("C:\\Models\\splade")`, `is_path_like("D:/foo")` (forward slash with drive), and `is_path_like("\\\\server\\share")`.

#### `cli::commands::infra::model::stop_daemon_best_effort` returns `false` on macOS while daemon is still running — model swap proceeds against live old-model daemon
- **Difficulty:** medium
- **Location:** `src/cli/commands/infra/model.rs:470-505` (`stop_daemon_best_effort`) and `src/cli/commands/infra/model.rs:509-549` (`restart_daemon_if_needed`).
- **Description:** `daemon_socket_alive` is `#[cfg(unix)]`-aware and correctly returns `true` on macOS when the daemon socket exists. But `stop_daemon_best_effort` only knows how to stop daemons via `systemctl --user`, gated `#[cfg(target_os = "linux")]`. On macOS the function takes the `#[cfg(not(target_os = "linux"))]` branch which logs `tracing::debug!("systemctl daemon control is Linux-only; skipping")` and **returns `false`**. The caller (`cmd_model_swap`) interprets `false` as "wasn't running" and proceeds with the in-place reindex. But the macOS daemon is still running — it still holds the OLD model in memory, still has the OLD store pool open, and is concurrently serving queries against the database file that the swap is rewriting. The result is data corruption risk on macOS during model swap: the daemon can serve stale embeddings, the OS-level file replacement may fail with EBUSY mid-swap, or the post-swap restart never happens because `restart_daemon_if_needed(false, ...)` early-returns. macOS is in the release target list (Linux + macOS ARM64 + Windows x86_64).
- **Suggested fix:** Detect any unix daemon by socket presence, then issue a stop signal that works without systemd. On macOS, read the PID from the socket lock file (`open_lock_file`/`write_pid` already write the daemon PID under `cqs_dir`), then `libc::kill(pid, libc::SIGTERM)`. Add an `#[cfg(target_os = "macos")]` branch alongside the existing Linux branch that does this. For a portable abstraction, factor a `daemon_pid(cqs_dir) -> Option<u32>` helper into `daemon_translate.rs`. Restart on macOS uses the same `cqs watch --serve` command via `Command::new` (no systemctl). The function's name is `best_effort` — that excuses Linux failure modes, not silent no-ops on a tier-1 platform.

#### `dispatch::try_daemon_query` deserializes `output` strictly as a string — a future daemon emitting structured `output` (per `daemon_translate::daemon_ping` design) silently falls back to CLI
- **Difficulty:** easy
- **Location:** `src/cli/dispatch.rs:739-741` (`if status == "ok" { return Some(resp.get("output")?.as_str()?.to_string()); }`) vs `src/daemon_translate.rs:275-285` (`daemon_ping` matches `output` as either `String` OR generic `Value`)
- **Description:** PR #1038 added `daemon_translate::daemon_ping` whose deserializer explicitly handles the case where the daemon writes `output` as a structured JSON value rather than a JSON-string-encoded payload (the comment at line 275-278 says "future-proofing if the envelope is ever changed"). But `try_daemon_query` — the canonical client-side dispatch path used by every other command — is hard-coded to `as_str()` and silently falls through to None on any non-string output. Any future daemon enhancement that returns structured output for non-ping commands (the obvious next step is `cqs ping --metrics` returning a non-string payload, or `cqs stats` returning a JSON object instead of pre-serialized string) breaks `try_daemon_query` with `tracing::debug!(stage = "parse", ...)` and a silent CLI fallback. #### `parser/chunk::extract_doc_fallback_for_short_chunk` strips `\r` from preceding lines but `lines()` already discards CRLF — strip is no-op for CRLF Windows files
- **Difficulty:** easy
- **Location:** `src/parser/chunk.rs:306` (`let lines: Vec<&str> = prefix.lines().map(|l| l.trim_end_matches('\r')).collect();`)
- **Description:** PR #1040 added `.map(|l| l.trim_end_matches('\r'))` to handle CRLF input (the audit-prep note flagged "verify no off-by-one on CRLF byte positions"). But `str::lines()` already documents that it splits on `\n` AND strips the optional preceding `\r` — the iterator yields `"foo"` from input `"foo\r\nbar\r\n"`, never `"foo\r"`. The `trim_end_matches('\r')` therefore only fires for sources containing bare `\r` (no following `\n`), which is the classic Mac OS line ending convention abandoned in 2001. So the line is doing nothing for the Windows CRLF case it was added for, and a maintainer reading the function will assume it correctly handles CRLF when in fact `str::lines()` is doing all the work. There's no off-by-one on CRLF byte positions because no CRLF byte arithmetic actually happens here — `prefix.lines()` returns `&str` slices that are already CRLF-stripped before the `trim` runs. Net effect: misleading dead code that simulates Windows-platform safety without providing it.
- **Suggested fix:** Either (a) remove the `.map(|l| l.trim_end_matches('\r'))` and add a comment "`str::lines()` already strips trailing `\r` from `\r\n` line terminators"; or (b) keep the trim and update the comment to clarify it covers bare-`\r` (classic Mac) input only, not Windows CRLF. Option (a) is cleaner. While there, add a unit test asserting `extract_doc_fallback_for_short_chunk` produces identical output for an input with `\n`-only line endings vs the same input with `\r\n` endings.

#### `enumerate_files` `to_ascii_lowercase` extension matching allocates one `String` per walked file
- **Difficulty:** easy
- **Location:** `src/lib.rs:518-527` (the extension filter `let lower = ext.to_ascii_lowercase(); extensions.contains(&lower.as_str())`)
- **Description:** The walker filters by extension via `ext.to_ascii_lowercase()` and matches against a registry of supported extensions. Windows filesystems are case-insensitive (NTFS preserves case but matches case-insensitively); macOS HFS+/APFS is case-insensitive by default; ext4 is case-sensitive. The current code allocates a fresh `String` per walked file just to do the case-insensitive comparison — for a 100k-file checkout that's 100k transient allocations per `cqs index`. Worse, `to_ascii_lowercase` is a no-op for non-ASCII bytes while Windows would still match case-insensitively (using full Unicode case folding on NTFS), producing a Windows/Linux divergence in which files get indexed if any future language registration introduces a non-ASCII extension. Every other case-comparison in cqs (e.g. `language::REGISTRY` lookups) uses `eq_ignore_ascii_case` rather than allocating a lowercased copy.
- **Suggested fix:** Replace `let lower = ext.to_ascii_lowercase(); extensions.contains(&lower.as_str())` with `extensions.iter().any(|e| ext.eq_ignore_ascii_case(e))`. No allocation per file, and the semantics match what the registry is documented as supporting. While there, add an integration test fixturing a `Foo.RS` and `Bar.Md` to assert both are picked up identically on case-insensitive (NTFS, APFS) and case-sensitive (ext4) filesystems.

#### `lib::normalize_path` does not strip Windows `\\?\` UNC prefix that `dunce::canonicalize` may return
- **Difficulty:** easy
- **Location:** `src/lib.rs:339-348` (`pub fn normalize_path(path: &Path) -> String` and `pub fn normalize_slashes(path: &str) -> String`)
- **Description:** `dunce::canonicalize` is supposed to return paths without the `\\?\` Windows extended-length prefix — that's its primary value over `std::fs::canonicalize`. But there are paths where `dunce` cannot remove the prefix: paths longer than `MAX_PATH` (260 chars), paths to files on volumes mounted via volume GUID (`\\?\Volume{GUID}\...`), and paths whose components include reserved-name characters that NTFS otherwise rejects (e.g. `\\?\C:\path\with\trailing dot.\file`). In those cases `dunce::canonicalize` returns the path with the `\\?\` prefix preserved. `normalize_path` then converts each `\` to `/`, producing `//?/C:/path/...` — which is neither a valid path on disk nor anything the rest of cqs treats consistently (`is_under_wsl_automount` at `cli/watch.rs:1112` would treat `//?/` as an unrecognized prefix; the staleness store keys the chunk by this string so future-lookup is by `//?/...` literal). The watch loop at `cli/watch.rs:1914-1916` then does `norm_path.starts_with(&norm_cqs)` to skip `.cqs/` events; if `cqs_dir` was canonicalized through this path the comparison silently fails. Database lookups orphan chunks because the same file is stored once as `//?/C:/Projects/cqs/src/foo.rs` and queried as `C:/Projects/cqs/src/foo.rs`. This is invisible on developer-machine smoke testing because typical project paths are well under 260 chars, but trips on the long-path case (deep monorepos, automated build outputs).
- **Suggested fix:** Strip the `\\?\` prefix in `normalize_path` (and `normalize_slashes` for the string variant) before the slash conversion. Concrete: `let s = path.to_string_lossy(); let s = s.strip_prefix("\\\\?\\UNC\\").map(|r| format!("\\\\{}", r)).unwrap_or_else(|| s.strip_prefix("\\\\?\\").unwrap_or(s.as_ref()).to_string()); s.replace('\\', "/")`. Apply identically to `normalize_slashes`. Add a unit test pinning `normalize_path(Path::new(r"\\?\C:\foo\bar"))` returns `"C:/foo/bar"` and `normalize_path(Path::new(r"\\?\UNC\server\share\foo"))` returns `"//server/share/foo"`.

#### `Store::is_slow_mmap_fs` is `#[cfg(unix)]`-only — Windows network drives (SMB / OneDrive / mapped Z:\) silently take 256 MB mmap × 4 conns and stall
- **Difficulty:** medium
- **Location:** `src/store/mod.rs:371-405` (`fn is_slow_mmap_fs`) — the `#[cfg(not(unix))]` branch returns `false` unconditionally.
- **Description:** The slow-mmap detection (added because WSL `/mnt/c/` 9P stalls SQLite mmap) reads `/proc/self/mountinfo` on Linux. On Windows the function returns `false` for every path, meaning `cqs index` running against `Z:\share\repo` (a mapped SMB share), `\\server\share\repo` (a UNC path), or any OneDrive-synced folder still requests the default 256 MB mmap × 4 connections (1 GB virtual address space). SMB doesn't support memory-mapped I/O cleanly — SQLite either fails the `MAP_PRIVATE` mmap entirely (silently falling back to read+write but logging only at debug) or the mapping appears to succeed but every page fault triggers a network round trip, making `cqs index`/`cqs search` dramatically slower than on local NTFS. Windows is a tier-1 release target and "user has their git checkout on a network share" is a common enterprise pattern, especially in the manufacturing/industrial context per the user memory.
- **Suggested fix:** Add a Windows arm to `is_slow_mmap_fs`. Use `GetVolumeInformationW` from `windows::Win32::Storage::FileSystem` to query the filesystem name; treat `"NTFS"` and `"ReFS"` and `"FAT32"`/`"exFAT"` (local) as fast and any other (`"SMBFS"`, `"NFS"`, `"CIFS"`, `"WebDAV"`) as slow. Alternative: detect remoteness via `GetDriveTypeW` on the volume root (`DRIVE_REMOTE` = SMB-class). For OneDrive/Dropbox sync folders (which present as local NTFS but are reparse-points to cloud blobs), check `GetFileAttributesW` for `FILE_ATTRIBUTE_REPARSE_POINT` and conservatively flag as slow. The default mmap size when slow is 0 (matching the existing Linux/WSL behavior) — disables mmap entirely on the slow path.

#### `ctrlc::set_handler` SIGINT-only — `cqs watch` on native Windows cannot be cleanly stopped, hard kill risks DB corruption
- **Difficulty:** hard
- **Location:** `src/cli/signal.rs:27-37` (single `ctrlc::set_handler` for both flag and force-exit) and `src/cli/watch.rs:118-132` (`install_sigterm_handler` is `#[cfg(unix)]`).
- **Description:** The `ctrlc` crate (without the `termination` feature) only traps SIGINT (Ctrl+C in a TTY). On Windows, `cqs watch` cannot be cleanly stopped by:
  - `taskkill /PID <pid>` — sends `WM_CLOSE`, ignored by a console app with no message loop
  - `taskkill /F /PID <pid>` — TerminateProcess, hard kill, no flush, possibly mid-write to `index.db` causing WAL corruption
  - Stopping a Windows Service wrapping `cqs watch` — uses `SERVICE_CONTROL_STOP`, no SIGINT equivalent
  Only an interactive Ctrl+C from the launching console works. The Linux-side SIGTERM handler (`install_sigterm_handler` at watch.rs:118-132) is `#[cfg(unix)]` so doesn't exist on Windows. This is the inverse of the macOS daemon-stop bug above: on Linux/macOS the daemon stops cleanly via SIGTERM; on Windows there is no clean stop path at all, only a hard kill that risks DB corruption. WSL deployments don't hit this because they stop via `systemctl --user`; native Windows users (which is the whole point of shipping windows-x86_64) do.
- **Suggested fix:** Either (a) enable the `ctrlc` `termination` feature so `Ctrl+Break` and console-close events are also trapped, plus document that "on Windows, send Ctrl+Break (`taskkill /PID <pid> /CTRL+BREAK`-equivalent or `Stop-Process -Name cqs`)" as the clean shutdown method; or (b) on Windows install a `SetConsoleCtrlHandler` (`windows::Win32::System::Console::SetConsoleCtrlHandler`) handler that catches `CTRL_CLOSE_EVENT` / `CTRL_LOGOFF_EVENT` / `CTRL_SHUTDOWN_EVENT` in addition to `CTRL_C_EVENT` / `CTRL_BREAK_EVENT`, sets the `INTERRUPTED` flag, and returns within the 5-second OS-mandated handler time budget. Option (b) is the proper fix because (a) doesn't help service-wrapped deployments. Cross-reference with the `serve` flag's existing fallback warning at `cli/watch.rs:1250-1257`.
