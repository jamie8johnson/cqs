#### cqs_test_map / cqs_trace inputSchema advertises a settable `max_nodes`, but the daemon JSON-args path silently drops it and re-resolves the env default (no-op of the documented OOM-guard knob)
- **Severity:** medium
- **Difficulty:** easy
- **Location:** src/cli/batch/json_args.rs:490-506 (test_map_args_from_core), :508-522 (trace_args_from_core); cores src/cli/commands/graph/test_map.rs:42-43 and graph/trace.rs:43-44; handler re-resolution src/cli/batch/handlers/graph.rs:366,:383,:433
- **Description:** The MCP/daemon inputSchema for cqs_test_map is schema::<TestMapCoreArgs> and for cqs_trace is schema::<TraceCoreArgs> (tools.rs:231,:239), where TestMapCoreArgs/TraceCoreArgs alias graph/test_map.rs::TestMapArgs and graph/trace.rs::TraceArgs (graph/mod.rs:34,:37). Both cores declare `pub max_nodes: usize` with `#[serde(default = ...)]` and NO `#[schemars(skip)]` (test_map.rs:42-43, trace.rs:43-44), so `max_nodes` is a published, settable inputSchema property. build_batch_cmd deserializes it into the core via parse_core::<TestMapCoreArgs>/<TraceCoreArgs> (json_args.rs:226,:233). But the converters test_map_args_from_core / trace_args_from_core then build args.rs structs (TestMapArgs{name,depth,cross_project,limit_arg}, TraceArgs{source,target,max_depth,cross_project,limit_arg}) that have NO max_nodes field — the deserialized c.max_nodes is read nowhere (json_args.rs:500-505, 513-521). The daemon handlers then re-resolve max_nodes unconditionally from the env: dispatch_test_map uses test_map_max_nodes() (graph.rs:366,:383) and dispatch_trace uses trace_max_nodes() (graph.rs:433). So a wire/MCP caller sending {"max_nodes":500} is silently ignored and the 10,000 env default is used. The converter comment at json_args.rs:493-495 frames this as consistent with argv ('max_nodes ... has no argv flag, so it does not round-trip'), which obscures that SETTING it over the wire — where the schema says you can — is a silent no-op. The json_args==argv parity test is structurally blind: parity_json_args_equals_argv's test-map/trace cases never set max_nodes (json_args.rs:730-735) and parity_json_args_with_explicit_fields covers callers/impact/dead only (json_args.rs:760-776), so both paths use the env default and match.
- **Suggested fix:** Either (a) thread max_nodes through: have dispatch_test_map / dispatch_trace honor the deserialized core's max_nodes instead of re-resolving the env in the handler (use c.max_nodes, with the env value only as the serde default fallback) — the core already carries the field; OR (b) if max_nodes is intentionally non-settable over the wire, add #[schemars(skip)] to the max_nodes field on both cores so it leaves the inputSchema, and tighten the doc comment to say the wire cannot set it. Add a json_args test that sets max_nodes and asserts the value reaches the core (argv==json-args parity cannot catch this since argv has no flag).
- **Verify verdict:** REAL. graph/test_map.rs:42-43 and graph/trace.rs:43-44 show `pub max_nodes: usize` with `#[serde(default)]`, no `#[schemars(skip)]`, on JsonSchema-deriving cores that ARE the cqs_test_map/cqs_trace inputSchema (tools.rs:231,:239 via graph/mod.rs:34,:37 aliases). json_args.rs:500-505 and :513-521 build args.rs structs with no max_nodes field, so c.max_nodes is dropped; graph.rs:366,:383,:433 re-resolve via *_max_nodes() ignoring the wire value. Parity tests (json_args.rs:730-735, 760-776) never set max_nodes, confirming the guard is blind.

#### MCP tools.rs keeps three hand-written `command → core type` maps (schema / validate / build) whose only parity guard checks command NAMES, not the type
- **Severity:** low
- **Difficulty:** medium
- **Location:** src/cli/mcp/tools.rs:155-259/:228-241 (read_tools input_schema: schema::<T>), :527-554 (validate_arguments check::<T>), vs src/cli/batch/json_args.rs:151-238 (build_batch_cmd parse_core::<T>); guard src/cli/mcp/mod.rs:148-165 + hardcoded list :99-105
- **Description:** The same command string is mapped to its Phase-0 core type T in three places that must agree: (1) the tool table rows set input_schema: schema::<T> (the advertised JSON Schema; tools.rs:155-259), (2) validate_arguments does check::<T> (the pre-relay shape check; tools.rs:527-554), (3) build_batch_cmd does parse_core::<T> (the authoritative daemon deserialize; json_args.rs:151-238). The only parity guard, tools_list_matches_json_args_registry (mod.rs:148-165), compares command NAME sets — exposed_commands() (names from tool_table, mod.rs:128-139) vs expected_read_commands() (a hardcoded name list, mod.rs:99-105). It does NOT pin that T is the same type across the three maps for a given command. The hardcoded json_args_capable array (mod.rs:99-103) is itself a fourth hand-maintained copy with a comment claiming it is 'mirrored from build_batch_cmd's match arms', but it is a literal, not derived — so the docstring claim that the guard pins the set against build_batch_cmd's arms is aspirational, not enforced. If a future refactor changed a command's core type in build_batch_cmd but left the old type in the schema/validate maps, the advertised inputSchema and the pre-check would silently disagree with the daemon's real deserialize, and every name-only guard stays green. validate_arguments also has no direct unit test (its only callsite is tools.rs:506), so a stale arm there is unguarded. Low severity because the daemon re-deserializes authoritatively (a wrong pre-check type causes a confusing false-reject/accept, not corruption), but it is the #1459 drift class on a brand-new surface.
- **Suggested fix:** Collapse the three maps to one source of truth: drive schema, validate_arguments, and build_batch_cmd from a single command → fn table (a const slice of (name, schema_fn, check_fn, build_fn), or a macro emitting all three arms from one row), so a command's type is declared once. At minimum add a guard test that for every tool command the type used by validate_arguments matches the one build_batch_cmd deserializes (round-trip a representative payload through both, assert both accept/reject identically), plus a direct per-arm test of validate_arguments.
- **Verify verdict:** REAL. Three separate type maps confirmed: input_schema: schema::<TestMapCoreArgs> etc. (tools.rs:231,:239,...), check::<TestMapCoreArgs> etc. (tools.rs:536-537,...), parse_core::<TestMapCoreArgs> etc. (json_args.rs:226,:233,...). Guard mod.rs:150-165 compares only name BTreeSets (exposed_commands mod.rs:128-139 vs expected_read_commands mod.rs:94-125); no type assertion. The 'mirrored from build_batch_cmd' list at mod.rs:99-103 is a literal array (fourth copy). grep shows validate_arguments has exactly one callsite (tools.rs:506) and no test.

#### cqs_index advertises a settable `slot` in its inputSchema but the daemon ignores it and returns queued:true with no divergence signal — a wrong-slot reindex request looks successful
- **Severity:** low
- **Difficulty:** easy
- **Location:** src/cli/commands/index/index_args.rs:37-46 (slot field, JsonSchema-derived) vs src/cli/batch/handlers/misc.rs:267-292 (dispatch_index)
- **Description:** The scoped IndexArgs core for cqs_index derives schemars::JsonSchema (index_args.rs:37) and exposes pub slot: Option<String> with no #[schemars(skip)] (index_args.rs:45), so cqs_index's inputSchema advertises slot as a settable parameter. dispatch_index reads args.slot only for the tracing span (misc.rs:273,:278) and calls ctx.request_reconcile() (misc.rs:276), which queues a reconcile of the slot the daemon was already opened against — args.slot never redirects the rebuild (also documented at index_args.rs:40-44 and misc.rs:258-261). The success envelope is {"status":"queued","queued":true,"reindex_deferred":true,"was_pending":...} (misc.rs:286-291) — no slot echo, no requested_slot_ignored flag. So an MCP client that calls cqs_index{"slot":"other"} to reindex a non-active slot gets an apparent success while the active slot (not the requested one) is what gets rebuilt, and the caller has no programmatic way to detect the no-op. The schema description is honest that slot is advisory (index_args.rs:40-44), so this is intended at the field level; the genuine defect is the silent success envelope on an agent-facing mutation surface — observability, not the advisory design itself.
- **Suggested fix:** Make the mismatch observable rather than silent: in dispatch_index, if args.slot is Some and differs from the served slot, either (a) return a handler error (cross-slot redirection unsupported on the fire-and-forget path), or (b) include the actually-reconciled slot and a requested_slot_ignored bool in the response so the caller can detect the divergence. Alternatively drop slot from the scoped core/schema entirely if cross-slot reindex is out of charter, matching the boundary-by-absence pattern the rest of IndexArgs follows.
- **Verify verdict:** REAL. index_args.rs:37 derives schemars::JsonSchema; :45 declares `pub slot: Option<String>` with no skip, so slot is in the advertised inputSchema. misc.rs:273,:278 read args.slot only for tracing; :276 calls request_reconcile() (no slot arg); :286-291 returns queued:true with no slot/ignored field. A {"slot":"other"} request therefore returns apparent success while the active slot is rebuilt, with no signal of the no-op. Advisory at the field level (documented), but the silent-success envelope is a genuine observability defect on a mutation surface.

#### MCP relay's 1 MiB response read cap silently truncates large-but-valid tool results and misreports them as a malformed-response protocol error
- **Severity:** medium
- **Difficulty:** medium
- **Location:** src/daemon_translate.rs:889-902 (daemon_json_args_request read cap) × src/cli/mcp/tools.rs:591-596 (bridge BadResponse mapping)
- **Description:** `daemon_json_args_request` — the sole MCP `tools/call` relay — reads the daemon response with `BufReader::new(&stream).take(1024 * 1024)` then `read_line` (daemon_translate.rs:892-897). The daemon writes the FULL response with no server-side size cap: handle_socket_client builds the response and calls write_daemon_ok → write_response_frame (socket.rs:404+) with no length bound; the only server cap is on the REQUEST line (socket.rs:122-146), not the response. When a legitimate tool response exceeds 1 MiB (plausible for cqs_gather/cqs_scout with token budgets, cqs_read of a large file, or cqs_search with many hits + content), the Take reader returns EOF exactly at the cap, so read_line returns Ok with a JSON line truncated mid-object. `serde_json::from_str(response_line.trim())` then fails (daemon_translate.rs:899) and the function returns `DaemonRpcError::BadResponse("parse envelope failed: ...")`. The bridge maps that variant (tools.rs:591-596) to JSON-RPC INTERNAL_ERROR `"cqs daemon returned a malformed response: ..."`. #### Parser walk-depth rail and parse-pool stack size are documented as a coupled load-bearing invariant but enforcement is one-directional — raising CQS_PARSER_MAX_WALK_DEPTH re-opens the stack-overflow SIGSEGV the rail prevents
- **Severity:** medium
- **Difficulty:** medium

- **Description:** The parser hardening pairs two knobs whose relationship the code explicitly documents as load-bearing: `PARSER_MAX_WALK_DEPTH = 800` is 'sized to complete inside a 1 MiB worker stack' (limits.rs:77-78, 92-93), and `PARSER_STACK_SIZE_MIN = 1 MiB` is the floor 'the depth rail is sized to fit' (limits.rs:102-105), with the comment calling the pairing a 'load-bearing-by-design invariant instead of an ambient assumption' (limits.rs:96-98). #### MCP bridge reads each stdin line unbounded before the 1 MiB cap — OOM on a no-newline line in the long-lived bridge process
- **Severity:** high
- **Difficulty:** easy
- **Location:** src/cli/mcp/bridge.rs:42-65
- **Description:** `serve_stdio` iterates `stdin.lock().lines()` (bridge.rs:42), which internally calls std's `read_line`/`BufRead::lines()` and appends the entire line into a fresh `String` with no upper bound. The 1 MiB cap is checked only AFTER the read, at `if line.len() > MAX_LINE_LENGTH` (bridge.rs:54), so it is post-hoc: the full bytes are already resident before the check can reject them. `cqs mcp` is a long-lived process — the `for line in stdin.lock().lines()` loop runs until stdin EOF/error (bridge.rs:42-92), entered via `cmd_mcp_dispatch` → `serve_stdio` (src/cli/commands/dispatch_shims.rs:233,242). A buggy or adversarial MCP host that writes a multi-GB chunk with no newline drives the bridge to allocate it all and OOM before the check fires — a memory-exhaustion primitive against the bridge. The sibling daemon-socket path does it correctly: `BufReader::new(&stream).take(take_cap)` plus `Ok(n) if n > max_request_bytes => TooLarge` (src/cli/watch/socket.rs:122-135) never materializes more than cap+1 bytes. The bridge is the straggler that did not adopt the bounded-read pattern.
- **Suggested fix:** Mirror the daemon's bounded read: replace `stdin.lock().lines()` with a manual loop over `BufReader::new(stdin.lock())` calling `read_line`/`read_until(b'\n', &mut buf)` on a reader wrapped in `.take(MAX_LINE_LENGTH as u64 + 1)`, so the read itself stops at the cap and an oversized line is rejected without first allocating it. Keep the post-read `len > MAX_LINE_LENGTH` check as a backstop emitting the PARSE_ERROR response. Add a regression test feeding a >1 MiB no-newline input and asserting bounded peak allocation / a clean PARSE_ERROR.
- **Verify verdict:** REAL. Confirmed at bridge.rs:42 (`for line in stdin.lock().lines()`) and bridge.rs:54 (post-read `if line.len() > MAX_LINE_LENGTH`) — std's read_line grows the String unboundedly before the cap check. The daemon's correct bounded read at socket.rs:124 (`.take(take_cap)`) and socket.rs:135 (`Ok(n) if n > max_request_bytes => ParseOutcome::TooLarge`) proves the asymmetry. Loop is long-lived (loops to EOF, bridge.rs:92; entered via dispatch_shims.rs:242). This is a NEW instance of the recognized 'cap rejects after the allocation' class — docs/audit-triage.md:70 (RM-V1.46.1-3) records the SAME class fixed for discover_delta in #1995, but the bridge is a recently-added (MCP Phase 1) site not covered by it; no bounded-read or oversized-line test exists in bridge.rs. Only citation slip: file path is src/cli/commands/dispatch_shims.rs, not src/cli/dispatch_shims.rs (line 242 is exact); does not affect the core defect at bridge.rs:42-65.
