## Error Handling

#### EH-V1.40-1: `filter_invoked_macros` LIKE pattern doesn't escape SQL wildcards
- **Difficulty:** easy
- **Location:** src/store/calls/dead_code.rs:178
- **Description:** `filter_invoked_macros` builds a LIKE pattern `format!("%{}!%", chunk.name)` from `chunk.name` and binds it to `WHERE content LIKE ?1`, but does NOT add `ESCAPE '\\'` clause and does NOT escape `_` / `%` / `\` in the macro name. SQL `LIKE` treats `_` as a single-char wildcard. Macro names commonly contain underscore (e.g. `define_languages`, `info_span`, `assert_eq`, `my_macro`) — every such macro's filter pattern matches more than intended. Concretely, a macro named `info_span` builds pattern `%info_span!%`, which matches content like `infoXspan!`, `info1span!`, etc., NOT just `info_span!`. Direction: false-positive matches inflate the "invoked" set, so legitimately dead macros are kept alive. Sibling code at `src/store/calls/mod.rs:197-211` already does the right thing: emits `ESCAPE '\\'` and uses `\_` for literal underscores. The new code drifted from that pattern. Bonus: the bound `chunk.name` is also susceptible to the operator's confusion if a macro name ever contains `%` literally — improbable but the LIKE pattern is unbounded.
- **Suggested fix:** Escape the literal `_`, `%`, and `\` characters in `chunk.name` before formatting, then add `ESCAPE '\\'` to the SQL: `let escaped = chunk.name.replace('\\', "\\\\").replace('_', "\\_").replace('%', "\\%"); let pattern = format!("%{}!%", escaped);` and `"... LIKE ?1 ESCAPE '\\\\' LIMIT 1"`. Add a regression test for `info_span` and `info1span`-content cases.

#### EH-V1.40-2: `Posture::current` silently picks Friendly on any non-`"1"` value of `CQS_ULTRASECURITY`
- **Difficulty:** easy
- **Location:** src/posture.rs:42
- **Description:** `Posture::current()` checks `std::env::var("CQS_ULTRASECURITY").as_deref() == Ok("1")` — any other value (`"true"`, `"yes"`, `"on"`, `"TRUE"`, `"1 "` with trailing space) silently falls back to `Friendly`, which omits the `handling_advice` advisory and downgrades the response shape used by adversarial deployments (cqs-as-MCP-server). The tests pin this contract (`current_reads_env_var` tests `"true"` → Friendly), but it's a footgun for the single most security-sensitive env var in the project: a typo silently disables ultrasecurity mode with no warning. `OutputFormat::current` (line 105) has the same problem — `=v2`, `=V1`, `=junk` all silently fall through to V2Bare. For `CQS_ULTRASECURITY` specifically, the silent disable is a documented-behavior bug per the `[Docs Lying Is P1]` memory rule: SECURITY/PRIVACY claims about adversarial-mode protection don't hold under common operator typos.
- **Suggested fix:** Recognize the common true-ish set (`"1"`, `"true"`, `"on"`, `"yes"`, case-insensitive) for Adversarial; for any value that's set but unrecognized, `tracing::warn!` once with the raw value and the resolved posture so the operator sees their typo. Same treatment for `CQS_OUTPUT_FORMAT`.

#### EH-V1.40-3: Trace `target` kind is not validated in either CLI or daemon path
- **Difficulty:** medium
- **Location:** src/cli/commands/graph/trace.rs:220-261, src/cli/batch/handlers/graph.rs:543-555
- **Description:** Both `cmd_trace` and `dispatch_trace` only run `classify_hits` against the **source** name and short-circuit Const/Type/Module/Ambiguous source kinds. The **target** name receives no kind dispatch — it falls straight into `resolve_target` and the BFS. If `target` is a Const/Type/Module/Ambiguous, the BFS searches for a path to a node that the call-graph can never contain, returning a "no path found" result that's indistinguishable from "the path actually doesn't exist." The user's mental model — "I asked for trace foo→Bar where Bar is a struct" — gets a false-negative empty result instead of "Bar is a Type, target name needs to be a function." Inconsistent contract: source asymmetry covered, target asymmetry not. Same plumbing exists (`store.lookup_by_name` + `classify_hits`); just need to apply it to target too.
- **Suggested fix:** After source kind dispatch, run an analogous `lookup_by_name` + `classify_hits` on the target name; if non-Function, emit a kind-labeled fallback (`{kind, fallback_from: "trace", source, target, target_kind, definitions, note}`) explaining the target isn't a function. Or, at minimum, log a `tracing::warn!(target, target_kind, "trace target is not a function — BFS will return empty")`.

#### EH-V1.40-4: `dispatch_deps` Type-forward semantics misroute Function names without warning
- **Difficulty:** easy
- **Location:** src/cli/commands/graph/deps.rs:198-203, src/cli/batch/handlers/graph.rs:155-178
- **Description:** Both `cmd_deps` and `dispatch_deps` treat `Function | Type | Multiple | Other | NotFound` as "fall through to existing flow." The existing flow then branches purely on the user-supplied `--reverse` flag: `--reverse=true` calls `get_types_used_by(name, ...)` (treats name as function); `--reverse=false` calls `get_type_users(name, ...)` (treats name as type). If the user runs `cqs deps my_function` (forward, default), but `my_function` is actually a Function (not a Type), `get_type_users("my_function")` returns an empty set silently — no warning that "you queried for a function but the forward branch expects a type name." The kind detection has the data needed to disambiguate (Kind::Function + reverse=false ⇒ user probably meant `--reverse`), but the code throws that signal away. Result: the user sees `[]`, falls through to grep, and the deps command becomes another datapoint in the "search-rate decline" the polymorphic-routing project is trying to fix.
- **Suggested fix:** In the Function/Type branches, cross-check `kind` against `reverse`. If `Kind::Function && !reverse`, emit a kind-labeled note suggesting `--reverse`. Mirror for `Kind::Type && reverse`. Pure additive — keeps the existing call but adds a guidance field.

#### EH-V1.40-5: `meta_value_for_envelope` and `meta_json_fragment_for_posture` fallback paths are unreachable but `.expect()` panics if they ever trigger
- **Difficulty:** easy
- **Location:** src/cli/json_envelope.rs:421
- **Description:** `meta_json_fragment_for_posture` calls `serde_json::to_string(&value).expect("Envelope meta serializes")` on the hot path of every JSONL daemon write. The `value` here is constructed from a `Map<String, Value>` of primitive types in the fallback path (line 401-419), which serde_json cannot fail to serialize today. But this is a hot-path `.expect()` in a daemon process — if a future schema change adds a field whose `Serialize` impl can return `Err`, every JSONL write will panic and crash the daemon. Bigger picture: the fallback at line 401 (`unwrap_or_else`) is dead defensive code (the original `to_value(&meta)` of `EnvelopeMeta`-of-primitives also can't fail today), and the second `.expect()` at line 421 has no fallback. Either the defense is real and the second call needs the same defensive `unwrap_or_else`, or both are dead and one should be removed.
- **Suggested fix:** Either remove the unreachable defensive fallback at line 401 and let the `expect` reflect the actual contract, or add a corresponding `unwrap_or_else` to the second `to_string` call. Pick one — the current shape (defended once, undefended once) is internally inconsistent.

#### EH-V1.40-6: `redact_error` chain_id collision via `DefaultHasher` is non-cryptographic
- **Difficulty:** easy
- **Location:** src/cli/json_envelope.rs:713-721
- **Description:** `redact_error` generates a chain ID by hashing `format!("{err:#}")` with `DefaultHasher` (SipHash). The doc comment says "Process-local; not a security boundary, just a grep target." That's fine for the grep-target use case, but `DefaultHasher` is randomized per-process (since Rust 1.36 on most targets) — the chain_id for the same error in two separate daemon invocations will not match. The implementation comment "retrying the same error gets the same id" only holds within a single process. Not a security bug, but the doc claim is wrong: operators correlating across daemon restarts will see different chain_ids for the same root cause. Worse, with 64-bit output reduced to 16 hex chars, the collision space within a single daemon run is fine, but the comment's "retrying" framing implies cross-process determinism that doesn't exist.
- **Suggested fix:** Either (a) update the doc comment to clarify "stable WITHIN a single daemon process" (cheap), or (b) switch to a deterministic hasher (e.g. `blake3::Hasher` or `std::hash::SipHasher13` with a fixed key) for cross-process determinism. Pick (a) unless cross-process correlation matters for ops.

#### EH-V1.40-7: `lookup_by_name` empty-string short-circuit silently differs from SQL semantics
- **Difficulty:** easy
- **Location:** src/store/chunks/query.rs:264-265
- **Description:** `lookup_by_name` returns `Ok(Vec::new())` for `name == ""` without an SQL roundtrip. The test at line 849 pins this. But chunks CAN have empty `name` strings — extracted unnamed entities (anonymous functions, lambdas, expression chunks) may carry `name = ""` in the chunks table. Real SQL semantics would surface those rows; the short-circuit suppresses them. The `kind::detect_kind_for_store` caller doesn't care (empty-name is an invalid query anyway), but if anything else calls `lookup_by_name("")` expecting "find every anonymous chunk," it gets an empty result with no indication that the query was rejected. Minor correctness drift between code and SQL semantics; documenting the shortcut would be enough.
- **Suggested fix:** Either (a) add a `tracing::debug!("lookup_by_name skipping empty-name fast path")` so the caller sees the divergence, or (b) document the shortcut in the rustdoc as "intentionally rejects empty-string names" so a future reader doesn't expect SQL-equivalent semantics.

#### EH-V1.40-8: `try_kind_fallback` daemon-path `lookup_by_name` SQL error surfaces as opaque internal error to client
- **Difficulty:** medium
- **Location:** src/cli/batch/handlers/graph.rs:84
- **Description:** Every daemon dispatch handler now calls `try_kind_fallback` first, which calls `ctx.store().lookup_by_name(name)?`. If that SQL query fails (table corruption, connection reset, `chunks` schema drift), the StoreError propagates up through anyhow → `redact_error` → daemon socket. The client sees `{"error": {"code": "internal", "message": "internal database error (err-...)"}}` with the chain_id. Pre-Phase 1, the same handlers would have hit a different SQL path (`get_callers_full`, `get_type_users`, etc.) — the error was the same, but it surfaced *after* the inline kind check did its work. Now every handler short-circuits on a single new SQL roundtrip *before* the existing flow runs, doubling the SQL surface that can fail. The handler doesn't distinguish "kind lookup failed (transient)" from "the actual graph query failed (transient)" — both collapse to the same redacted envelope. For long-running daemon processes this means a single transient SQLite contention event takes out every kind-routed command for a tick.
- **Suggested fix:** In `try_kind_fallback`, treat `lookup_by_name` SQL errors as "couldn't classify, fall through to existing handler" rather than fatal. Log via `tracing::warn!(error = %e, name, "Kind classification failed, falling back to existing handler")` and return `Ok(None)`. The existing handler may then succeed (it queries different tables) or fail with its own error. Keeps fragility additive instead of multiplicative.

#### EH-V1.40-9: `chunks_to_definitions` / `build_kind_fallback_value` echo full chunk content into JSON without size cap
- **Difficulty:** medium
- **Location:** src/cli/commands/graph/impact.rs:178-193, src/cli/batch/handlers/graph.rs:39-52
- **Description:** Every kind-mismatch fallback emits `"content": c.content` for every matched chunk verbatim. For `Kind::Ambiguous` (a name that resolves across multiple kinds), the response can include 5+ chunks, each with full content — easily 50KB+ on hot names that match function/method/const/macro/module variants. The CLI direct path is fine (operator sees it). The daemon JSONL path serializes one line per dispatched command; a 50KB line in the daemon output stream isn't an error per se, but it amplifies a noisy-name query into a daemon-side memory + bandwidth spike with no upper bound. The other graph commands (callers/callees/impact-function-path) all `truncate(limit)` before serialization. Kind-fallbacks don't truncate at all.
- **Suggested fix:** Apply the same `clamp(1, 100)` cap on the `chunks` slice at the start of each fallback dispatcher / `try_kind_fallback`. For very-large `c.content` (>2KB say), truncate to first N chars + `"... (truncated, full at file:line)"` marker. Mirror the size discipline that the rest of the graph commands ship with.

#### EH-V1.40-10: `dead_code::filter_invoked_macros` swallows individual macro lookup errors via `?` short-circuit
- **Difficulty:** easy
- **Location:** src/store/calls/dead_code.rs:166-192
- **Description:** `filter_invoked_macros` runs N sequential SQL queries (one per Macro candidate) inside a `for` loop, propagating each `await?` error to the caller via `?`. If any single macro's `LIKE` query fails — say, transient SQLite contention on chunk 47 of 100 — the entire dead-code analysis fails with one redacted internal error. Pre-#1623 the analysis had no per-macro filter step; failure modes were "the whole analysis ran or the whole analysis failed." Now there's a long-running serial loop of independent queries where one transient hiccup zeroes the whole result. Worse: the user-facing error message is just "internal database error" — it doesn't say "macro filter pass failed at macro #47, partial results would have been correct without filter." `find_dead_code` is also the kind of operation an operator runs once an hour; serializing every macro invocation check makes it brittle.
- **Suggested fix:** Convert the `?` to a `match` that logs `tracing::warn!(macro_name, error = %e, "Failed to query macro invocation, retaining as candidate")` and falls back to keeping the macro in the candidates list (false-negative-friendly direction the file doc-comment already endorses). Caller gets a slightly noisier dead-code list on transient SQL hiccup instead of total failure. Same pattern as `score_confidence`'s "content missing" warn-and-continue (line 304).

#### OB-V1.40-1: Kind-fallback dispatches emit no `tracing::info!` — Phase 2 prioritization signal is lost
- **Difficulty:** easy
- **Location:** `src/cli/commands/graph/callers.rs:106-148, 178-217, 293-336, 358-394`; `src/cli/commands/graph/deps.rs:76-104, 171-198`; `src/cli/commands/graph/test_map.rs:258-294, 336-380`; `src/cli/commands/graph/impact.rs:97-100, 303-396`; `src/cli/commands/graph/trace.rs:511+`; `src/cli/batch/handlers/graph.rs:78-104, 152-178, 227-245, 287-310, 359-385, 464-490, 543-570`
- **Description:** Eight kind-fallback dispatchers fire when an agent runs a function-shaped command (`callers`, `callees`, `impact`, `test-map`, `deps`, `trace`) against a name that classifies as `Const | Type | Module | Ambiguous`. Each builds a redirect payload and returns. **None emit a structured tracing event** at the moment of mismatch. The CLI path's `_span = tracing::info_span!("cmd_callers", ...)` covers the entry but doesn't distinguish the fallback branch from the happy path. The daemon path's `_span = tracing::info_span!("batch_callers", ...)` has the same shape — `try_kind_fallback` returns `Some(value)` and gets returned without any log line. With ~50% of agent traffic going through the daemon, the v1.40.0 hypothesis ("agents call `cqs impact CONST_FOO` and the kind-mismatch surfaces silently as `[]`") is unmeasurable from journald — the only discriminator would be parsing the JSON response shape, which telemetry doesn't capture either (see OB-V1.40-2).

The `dispatch_deps` inline kind dispatch at `:152-178` needs the same. Operators harvesting journald can then `grep '"kind-fallback"'` and bin by `(fallback_from, kind)` to see whether the v1.40.0 hypothesis holds.

#### OB-V1.40-3: `Posture::current()` and `OutputFormat::current()` log nothing when env var overrides default
- **Difficulty:** easy
- **Location:** `src/posture.rs:41-47, 104-110`
- **Description:** Both env-var-driven knobs (`CQS_ULTRASECURITY=1` for adversarial posture, `CQS_OUTPUT_FORMAT=v1` for legacy envelope) silently flip behavior with **zero tracing events**. SNR Phase 4 (this commit, per the doc-comment) just flipped the *default* of `OutputFormat`, so the `CQS_OUTPUT_FORMAT=v1` opt-out is now the load-bearing migration hedge for consumer scripts — and an operator who set the env var has no way to verify it took effect short of running a command and inspecting the wire shape. Compare to `trace_max_nodes()` at `cli/commands/graph/trace.rs:443-453` which emits `tracing::info!(cap = n, "Trace BFS node cap overridden via CQS_TRACE_MAX_NODES")` on override and `tracing::warn!(val, "CQS_TRACE_MAX_NODES invalid, ...")` on parse failure. Posture/OutputFormat have neither.
- **Suggested fix:** Convert both `current()` methods to use `OnceLock` (the env vars are stable across a process lifetime — daemon reads them once at boot). On `Posture::current()` first call, emit `tracing::info!("Adversarial posture active via CQS_ULTRASECURITY=1")` when the env triggers; emit nothing on the default Friendly path (no spam for the no-override case). On `OutputFormat::current()` first call, emit `tracing::info!("Legacy envelope active via CQS_OUTPUT_FORMAT=v1")` when the override triggers and a one-line `tracing::warn!(val = %v, "CQS_OUTPUT_FORMAT='{v}' is not 'v1' — using default V2Bare")` when the env is set to anything else (typo signal — matches the trace_max_nodes pattern).

#### OB-V1.40-5: `detect_kind_for_store` and `classify_hits` have no entry span — Phase 1 plumbing is invisible in tracing

#### OB-V1.40-6: `filter_invoked_macros` (Tier 2b) drops candidates silently — operator can't audit false-positive rate
- **Difficulty:** easy
- **Location:** `src/store/calls/dead_code.rs:166-192`
- **Description:** PR #1573 Tier 2b added macro-invocation filtering: each `Macro` candidate is re-checked with a `LIKE '%<name>!%'` content scan, dropped if any chunk references the bare invocation. The doc-comment is honest about the false-negative bias (a comment `// foo! is broken` keeps the macro live), but **the function never logs how many candidates it dropped**. The parent `find_dead_code` info event at `:76-81` reports `total_uncalled / confident / possibly_dead` but the macro-filter step is between phase 1 and phase 2 and contributes neither figure visibly. For a project with 100 declared macros, the operator sees "5 confident dead" without any signal that 95 macros were filtered — was the LIKE pattern matching too broadly (e.g. `assert!` is referenced everywhere, `log!` is everywhere) or correctly narrowly?
- **Suggested fix:** Track the filter delta and log: `let pre_count = candidates.iter().filter(|c| c.chunk_type == ChunkType::Macro).count();` before the filter, `let dropped = pre_count - filtered.iter().filter(|c| c.chunk_type == ChunkType::Macro).count();` after. Emit `tracing::info!(macro_candidates = pre_count, dropped_as_invoked = dropped, kept_as_dead = pre_count - dropped, "filter_invoked_macros: LIKE '%<name>!%' filter applied")` if `pre_count > 0`. Operators tuning the macro filter can correlate "the dead list has 0 macros" against "we filtered 95/95 — pattern is too eager".

#### OB-V1.40-7: `write_json_line` reads `Posture::current()` per-emission with no log — daemon posture decision unattributable
- **Difficulty:** easy
- **Location:** `src/cli/batch/mod.rs:2209-2286`
- **Description:** `write_json_line` is the daemon hot path: every JSON-line response goes through it, and it reads `Posture::current()` at line 2227 to decide between the slim Friendly envelope and the verbose Adversarial envelope. The two branches emit different bytes (line 2233 `error: null`/`version` is conditional; line 2241 `_meta` fragment is conditional). **Zero log line records the posture decision.** Combined with OB-V1.40-3 (no log on env-var read), an operator who upgraded the daemon binary mid-session or set `CQS_ULTRASECURITY=1` in shell after the daemon started has no way to verify the wire shape per-request. The `tracing::debug!` at line 2260 only fires on float-sanitize fallback. No info_span at entry — the function is leaf-level on the daemon path so it doesn't strictly need one, but the posture branch needs at least debug-level visibility.
- **Suggested fix:** Either (a) add `tracing::debug!(posture = ?posture, "write_json_line dispatching")` at line 2228 (cheap per-request, off by default but greppable when investigating "wire-format mismatch reports"), or (b) hoist the posture read out of `write_json_line` to the daemon's `BatchContext` initialisation (where it's read once and cached), then log at startup. Either way, the posture decision should be traceable from logs.

#### OB-V1.40-8: Daemon socket `accept()` success has no log — handle leaks unattributable
- **Difficulty:** easy
- **Location:** `src/cli/watch/daemon.rs:203-247`
- **Description:** The accept loop logs four things: rejection at concurrency cap (`tracing::warn!` line 216), spawn failure (`tracing::warn!` line 242), accept errors (`tracing::warn!` line 259), socket cleanup on drop (`tracing::info!` socket.rs:26). It does **not log successful accepts**. With `CQS_MAX_DAEMON_CLIENTS` defaulting to 16-64, a leaked handler thread (e.g. `handle_socket_client` blocked on a wedged store mutex) silently bumps `in_flight` without any log signal until the cap is hit and the next connection sees the rejection warn. #### CQ-V1.40-10: `filter_invoked_macros` issues N separate SQL queries (one LIKE per Macro candidate) — should batch
- **Difficulty:** easy
- **Location:** `src/store/calls/dead_code.rs:166-192`
- **Description:** Phase 1.5 of `find_dead_code` filters out macros invoked at file-scope (closes #1573 Tier 2b). For each Macro chunk in the candidate list, it issues a separate SQL query: `SELECT 1 FROM chunks WHERE content LIKE ?1 LIMIT 1` with pattern `%<name>!%`. This is an N+1 query pattern — N round-trips through `block_on(async)` per `cqs dead` invocation. On the cqs corpus, `gen_log_query_dispatch` is the only Macro currently surfaced (per the `cqs dead --json` output on this machine), so N=1 today; but the v1.7.0 dead-code disaster taught us this is exactly the kind of pattern that scales badly when the corpus grows or when a multi-language repo with heavy macro use indexes. The right shape is: build one IN clause with all macro names, do a single `SELECT name FROM macros WHERE EXISTS (SELECT 1 FROM chunks WHERE content LIKE '%' || name || '!%')` — or, since the LIKE is already O(corpus size) on `content`, scan all chunk content **once** and check every macro name against it in-memory.
- **Suggested fix:** Replace per-macro query with: (1) extract all Macro candidates' names into a `HashSet<String>`, (2) one `SELECT content FROM chunks` to stream all content, (3) for each chunk content scan, check every macro name (or build a regex `/(<name1>|<name2>|...)!/` and run once). Saves N-1 round trips. For the cqs index today this is a sub-millisecond difference but the algorithm is correct and the SQLite per-statement preparation cost is non-zero.

#### API-V1.40-2: `Kind` enum mixes 5 routing kinds with 3 resolution outcomes (Ambiguous/Multiple/NotFound)
- **Difficulty:** medium
- **Location:** `src/kind.rs:40-65`
- **Description:** `pub enum Kind { Function, Type, Const, Module, Other, Ambiguous, Multiple, NotFound }` collapses two orthogonal concepts into one type. The first 5 variants describe what kind of definition a chunk *is* (consumed by `classify_chunk_type(ChunkType) -> Kind`). The latter 3 describe the *resolution outcome* of a multi-hit lookup (only ever produced by `classify_hits`). `classify_chunk_type` provably never returns Ambiguous/Multiple/NotFound — but every consumer of `Kind` (every `match kind { ... }` in the 6 graph command dispatchers, the batch handler, and the test_map dispatcher) has to enumerate all 8 arms or risk a compile-time non-exhaustive match. Result: the ergonomic move ("dispatch on Kind to pick a fallback") sees `Kind::Multiple` and `Kind::NotFound` mixed in with the legitimate kinds, and every match arm spells "this branch is unreachable from this caller" by hand. The doc comment at line 37 acknowledges the mix ("`NotFound`, `Ambiguous`, and `Multiple` are dispatch decisions rather than chunk-type groupings") but the type doesn't enforce the split.
- **Suggested fix:** Split into two types: `pub enum Kind { Function, Type, Const, Module, Other }` (pure classification — what `classify_chunk_type` returns) and `pub enum KindResolution { Ambiguous, Multiple, NotFound, Single(Kind) }` (what `classify_hits` returns). Match arms on the dispatcher side become 4 explicit cases instead of 5+3 with implicit unreachables. `classify_chunk_type` is now total — no caller has to handle a phantom `Kind::Ambiguous` it could never observe.

#### API-V1.40-6: `emit_json_error_with_data_and_posture` ignores posture when shaping the envelope, contradicting `wrap_error_with_posture`

#### API-V1.40-7: BatchProvider trait method names use 6 different verbs for closely related operations
- **Difficulty:** medium
- **Location:** `src/llm/provider.rs:121-188`
- **Description:** Method naming inside the same trait:
  ```
  submit_batch(...)           // verb_noun
  check_batch_status(...)     // verb_noun_X
  wait_for_batch(...)         // verb_for_noun
  fetch_batch_results(...)    // verb_noun_X
  is_valid_batch_id(...)      // is_adj_noun_X (predicate)
  model_name(...)             // noun_noun (getter)
  validate_model(...)         // verb_noun (different return shape from is_valid_batch_id!)

The `_batch_` infix appears in 4 of 7 method names but not consistently; `is_valid_batch_id` would naturally be `is_valid_id` (the trait's whole subject is batches). `model_name` returns `&str` but `validate_model` takes `&str` — the round-trip `provider.validate_model(provider.model_name())` is the canonical shape and the docstrings say so, yet the method names don't telegraph it.

#### RB-V1.40-2: No index on `chunks.name` — every polymorphic-routing call is full table scan
- **Difficulty:** medium
- **Location:** `src/store/chunks/query.rs:262-279` (`lookup_by_name`); call sites `src/cli/commands/graph/{impact.rs:93, callers.rs:174,354, deps.rs:167, test_map.rs:254, trace.rs:220}` and `src/cli/batch/handlers/graph.rs:84, 152` (8 production callers)

#### RB-V1.40-3: `classify_hits` uses `.expect()` for a provably-safe path — violates project's "no unwrap outside tests" rule
- **Difficulty:** easy
- **Location:** `src/kind.rs:177`
- **Description:** `classify_hits` ends with `*kinds.iter().next().expect("non-empty single-kind set")`. The expect is logically safe: at this point `kinds.len() == 1` is guaranteed by the preceding `if kinds.len() > 1` early-return and the `if hits.is_empty()` guard. The `.expect()` will never fire under correct inputs. **But** CLAUDE.md states: "No `unwrap()` except in tests" — and `expect()` carries the same panic semantics. The lib crate uses `thiserror`, not anyhow, and a panic in this code path crashes whatever process calls `detect_kind_for_store` (CLI direct, daemon, batch). Adding a new `ChunkType` variant in the future + missing it in `classify_chunk_type` could route into the `Kind::Other` arm in a way that still preserves the "kinds.len() == 1" invariant, but a refactor that swaps the early-return order or splits classify_hits into multiple functions could break the invariant without the compiler catching it. Cheap to express via a non-panicking shape.
- **Suggested fix:** Replace the unreachable arm with an explicit kind-or-fallback:
  ```rust
  match kinds.into_iter().next() {
      Some(k) => k,
      None => {
          tracing::warn!(hits = hits.len(), "classify_hits: empty kinds set despite non-empty hits — treating as Other");
          Kind::Other
      }
  }
  ```
  Same observable behavior on the happy path, no panic surface. Or use `kinds.iter().copied().next().unwrap_or(Kind::Other)` if the warn is too noisy. The latter is one line.

#### RB-V1.40-4: `filter_invoked_macros` is N+1 SQL — 10k macros = 10k roundtrips with one `LIMIT 1` query each
- **Difficulty:** medium
- **Location:** `src/store/calls/dead_code.rs:166-192`
- **Description:** PR #1573 Tier 2b added macro-invocation filtering. The implementation is a serial for-loop with one SQL query per Macro candidate:
  ```rust
  for chunk in candidates {
      if chunk.chunk_type == ChunkType::Macro {
          let pattern = format!("%{}!%", chunk.name);
          let row: Option<(i64,)> = sqlx::query_as("SELECT 1 FROM chunks WHERE content LIKE ?1 LIMIT 1")
              .bind(&pattern).fetch_optional(&self.pool).await?;
          ...
      }
  }
  ```
  The function doc-comment explicitly claims "fast even on large indexes" via the `LIMIT 1` short-circuit. That's true *per query*, but at 10k macro candidates the round-trip overhead alone (sqlx parameter bind + content scan + LIMIT 1 stop + return) is ~1-3ms × 10000 = 10-30 seconds added to every `cqs dead` run. Most agents run dead-code analysis once per session, so the cost is bounded — but the latency is highly visible in the user-facing flow. Worse, the macro count grows linearly with codebase size, so scaling up corpus size scales up dead-code wall time multiplicatively (corpus growth × macro-density). On a 100k-chunk corpus with 5k macros, 15 seconds added.
- **Suggested fix:** Convert to a single `OR`-concatenated LIKE query (with proper escaping per RB-V1.40-5 below):
  ```rust
  let macros: Vec<&LightChunk> = candidates.iter().filter(|c| c.chunk_type == ChunkType::Macro).collect();
  if macros.is_empty() { return Ok(candidates); }
  // Build: SELECT name FROM chunks_with_macro_invocations WHERE EXISTS ...
  // Or: per-batch query with chunked OR (LIKE ?1 OR LIKE ?2 OR ... LIKE ?N) and a name-binding pass.
  ```
  Or precompute: extract every `\bword!` token from every chunk's content into a temporary set, then check macro names against that set in O(1). One full-content scan instead of one per-macro scan. Or — the cheapest middle ground — restructure as a single SQL that joins the candidate names against chunks.content via a temporary CTE: `WITH names AS (VALUES (?), (?), ...) SELECT name FROM names WHERE EXISTS (SELECT 1 FROM chunks WHERE content LIKE '%' || name || '!%')`. One roundtrip per ~999 candidates instead of one per candidate.

#### AC-V1.40-1: `filter_invoked_macros` LIKE pattern requires `!` suffix — every C/C++ `pp_define`, Elixir `defmacro`, Julia `macro_definition` is misclassified as dead
- **Difficulty:** medium
- **Location:** src/store/calls/dead_code.rs:178
- **Description:** `filter_invoked_macros` builds `format!("%{}!%", chunk.name)` and only matches content where the macro name is followed by `!`. That convention is Rust-specific (`macro_rules!` invocations only). The same `ChunkType::Macro` enum is set by:
  - `language/languages.rs:1483` for C/C++ `pp_define` (#define MY_MACRO) — invoked as `MY_MACRO(args)`, no `!`.
  - `language/languages.rs:3477` for Julia `macro_definition` — invoked as `@my_macro x`, no `!`.
  - Elixir `defmacro` — invoked as `Module.my_macro(args)`, no `!`.
  - Rust `macro_rules!` derive macros and proc macros (when classified as Macro) — invoked as `#[derive(MyMacro)]` or `#[my_macro]`, no `!` either.
  
  Net effect: the Tier 2b filter only fires for Rust function-style macros. Every C/C++ project has every `#define` reported as dead-code; same for Elixir/Julia macro-heavy codebases. Comment at line 173-177 incorrectly claims "even the bare name typically appears with `!`, so the suffix is the right discriminator" — that's only true in Rust function-macro syntax. The doc-comment for the function (line 156-165) frames this as "Closes the file-level macro-invocation gap (#1573 Tier 2b)" but it's a Rust-only fix that systematically false-positives every other language with macros.
- **Suggested fix:** Branch on `chunk.language` — for Rust use `%{}!%`, for C/C++/Cython use `%{}(%` (or just `%{}%` since the JOIN-against-callee_name in `fetch_uncalled_functions` already handles direct call-graph edges and the gap is specifically file-scope invocations). Alternatively run two patterns and OR them — the cost is one extra LIKE per Macro candidate, dwarfed by the existing N+1 SQL pattern.

#### PB-V1.40-9: WAL mode + WSL DrvFS: cqs unconditionally enables `journal_mode=WAL` on every store open, but SQLite WAL is unsupported on `/mnt/c/` (9P bridge)
- **Difficulty:** medium
- **Location:** src/store/mod.rs:944 (`.journal_mode(SqliteJournalMode::Wal)` unconditional); cache.rs:291 (same); both never check `is_wsl_drvfs_path` first
- **Description:** SQLite's WAL journal mode requires shared memory via mmap of the `-shm` sidecar file, and explicitly documents that WAL mode "does not work over a network filesystem" (per https://sqlite.org/wal.html §2.2). - **Suggested fix:** Detect at open-time: `if cqs::config::is_wsl_drvfs_path(path) { connect_opts.journal_mode(SqliteJournalMode::Delete) }`. #### SEC-V1.40-7: Daemon socket request reading uses `BufRead::read_line` which doesn't enforce the `take(N)` cap on the FIRST line — multi-MB single-line request reaches the parse path
- **Difficulty:** medium
- **Location:** src/cli/watch/socket.rs:115-135 (`read_line` against `BufReader::new(&stream).take(take_cap)`)
- **Description:** The daemon caps request size via `BufReader::new(&stream).take(take_cap)` where `take_cap = max_request_bytes + 1`. The intent: requests larger than `max_request_bytes` should hit the `Ok(n) if n > max_request_bytes ⇒ TooLarge` arm at line 129. Read carefully: `BufRead::read_line` reads UNTIL a newline or EOF. If the client sends a 10 MB request without a newline followed by EOF, `read_line` returns `Ok(10_000_000)` only if the underlying reader returns the full 10 MB before EOF. The `Take<R>` adapter caps the bytes at `take_cap`, so `read_line` returns at most `take_cap` bytes, and the `n > max_request_bytes` check correctly fires at `n == max_request_bytes + 1`. So far so good. **The problem**: between when `Take` returns 0 bytes (EOF-by-cap) and `read_line` returns `Ok(n)`, the entire `take_cap` bytes have been *allocated* into `line: String`. A malicious client sending 5 MB of garbage (no newline) at the daemon causes a 5 MB string allocation per connection, multiplied by `CQS_MAX_DAEMON_CLIENTS` (default 16, max 64) parallel connections — 80 MB → 320 MB transient memory pressure on a misbehaving fan-out, with no per-connection memory cap below the `take_cap`. Given the daemon is local-only, threat model is "misbehaving agent / accidental fan-out," not network adversary; but the cap is the wrong tool for that mitigation. The real cap should be "fail-fast on first newline-free chunk over a threshold." Pre-#1463 the cap was hardcoded 1 MB; #1463 raised it to `max_diff_bytes() + 4096` ≈ 5 MB. The raise inflated the worst-case allocation per connection 5×.
- **Suggested fix:** Either (a) cap `String::with_capacity` at a smaller initial value (8192 — the existing line value at line 107) and let `read_line` grow incrementally; the heap pressure tracks the actual line shape rather than a worst-case 5 MB pre-allocation, OR (b) read in chunks via `BufRead::read_until(b'\n', ...)` against a 64 KB scratch buffer with an explicit byte counter that bails on overshoot. (a) is cheaper and matches `String`'s amortized-doubling growth contract; (b) is more defensive but a larger refactor. Either fix should land with a regression test that pumps 10 MB of newline-free garbage at the socket and asserts the handler returns within a memory budget commensurate with the actual data.

The doc-comment claim is a `[Docs Lying Is P1]` violation. The destructive migrations (v18→v19 drops `sparse_vectors`, v22→v23 hard column-rename) are exactly the cases where this matters most.
- **Suggested fix:** Before the triple-restore, delete the live DB's existing `-wal` / `-shm` (so SQLite doesn't replay them against the restored main), then restore the main DB, then restore sidecars. If the process dies anywhere in this sequence, SQLite's next open sees a restored main + missing/partial sidecars → it creates fresh sidecars and the restored main is canonical. The current order has the inverse problem: the live `-wal` (from the failed migration) survives the restore and contaminates the recovered state. Add a regression test that injects a kill between the main-DB rename and the WAL rename, then opens the DB and asserts the schema_version reflects the pre-migrate state.

#### PERF-V1.40-1: `filter_invoked_macros` LIKE `%name!%` is unindexable full-table scan — quadratic in (macros × chunks)
- **Difficulty:** medium
- **Location:** src/store/calls/dead_code.rs:178-183
- **Description:** Already-reported CQ-V1.40-10 / RB-V1.40-4 / DS-V1.40-6 cover the N+1 angle (one query per macro). The `chunks` table contains 14k rows on cqs main (per `cqs health`), each with multi-KB `content`. The right shape is "build the dictionary once, scan once" not "scan per macro." Adds index dependency but eliminates the quadratic. Even just batching into `WHERE content LIKE ? OR content LIKE ? OR ...` (single scan) collapses N scans to 1.

#### PERF-V1.40-9: Telemetry write does ~5 syscalls per CLI command (open lock, try_lock, stat path, open path, write) — file handles never reused

#### PERF-V1.40-10: Tier 2a `field_initializer (identifier)` query inflates `function_calls` table — every struct-literal `field: var` produces a row
- **Difficulty:** medium
